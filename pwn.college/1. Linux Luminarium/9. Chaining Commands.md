
### Chaining with semicolons (`;`)

The easiest way to chain command is `;`. In most contexts, `;` separates commands in a similar way to how Enter separates lines. So, this:

```console
hacker@dojo:~$ echo COLLEGE > pwn
hacker@dojo:~$ cat pwn
COLLEGE
hacker@dojo:~$
```

Is roughly the same as this:

```console
hacker@dojo:~$ echo COLLEGE > pwn; cat pwn
COLLEGE
hacker@dojo:~$
```

### Building on success (`&&` chaining)

If you want the first command to run successfully (exited with code 0) as a prerequisite to running the second command, you can chain the commands with `&&`. `&&` acts like an `AND` operator in the way that both conditions must be true; the first command must succeed then the second command will run. This is useful in real life workflows such as certain actions needing to depend on success of other actions.

```console
hacker@dojo:~$ command1 && command2
```

This means, run command1, and if it succeeds, run command2.

Some examples:

```console
hacker@dojo:~$ touch /home/hacker/file && echo "this will run"
success
this will run
hacker@dojo:~$ touch /file && echo "this will NOT run"
touch: cannot touch '/file': Permission denied
hacker@dojo:~$
```

The second invocation of `touch` failed because the hacker does not have the write access to `/file`, so `echo` did not run.

### Handling failure (`||` chaining)

Opposite of `&&`, `||` works in a way that the second command only runs if the first command fails, similar to `OR` operation. This means that either the first command succeeds OR the second command will run.

```console
hacker@dojo:~$ command1 || command2
```

This means that "Run command1, and if it fails, run command2"

Some examples:

```console
hacker@dojo:~$ touch /file || echo "touch failed, so this runs"
touch: cannot touch '/file': Permission denied
touch failed, so this runs
hacker@dojo:~$ touch /home/hacker/file || echo "this will NOT run"
hacker@dojo:~$
```

The `||` operator is very useful in cases such as error handling or fallback commands.

### Shell scripting

When you need multiple commands that run sequentially, a simple chaining such as `;`, `&&`, or `||` would not suffice. If you need multiple lines of commands to execute, you better use a script. We can create a shell script with `<script_name>.sh`.  For example, we can create a shell script called `pwn.sh`. By convention, shell scripts are frequently named with a `sh` suffix.

```sh
echo COLLEGE > pwn
cat pwn
```

We can then execute them by passing the script name as argument to a new instance of our shell (`bash`). When a shell is invoked like this, rather than taking commands from user, it reads commands from the file.

```console
hacker@dojo:~$ ls
hacker@dojo:~$ bash pwn.sh
COLLEGE
hacker@dojo:~$ ls
pwn
hacker@dojo:~$
```

Running `bash` always is a pain. To avoid having to write `bash` when executing script every time, you can `chmod` the script to be executable:

```console
mayo@localhost:~> cat hello.sh    
echo "hello, world!"  
mayo@localhost:~> ls -l hello.sh    
-rw-r--r-- 1 mayo mayo 21 Apr 16 11:06 hello.sh  
mayo@localhost:~> chmod +x hello.sh    
mayo@localhost:~> ls -l hello.sh    
-rwxr-xr-x 1 mayo mayo 21 Apr 16 11:06 hello.sh  
mayo@localhost:~> ./hello.sh    
hello, world!  
mayo@localhost:~>
```

### Shebangs

Before you consider yourself a shell scripter, you must understand what are shebangs. So far, *your shellscripts can only be run from the shell*. If you want to invoke your script from a python program, things won't work well compared to what we expected.

When a program is invoked in Linux, the Linux kernel first inspects the file to determine how it should be run. This does NOT use the extension (which is why you don't have to explicitly name your shellscripts in `sh` extension, or python programs in `py` extension). Rather, Linux looks at the first few bytes of the file for this information.

If a program starts with a shebang (characters `#!`), Linux treat the file as *interpreted program*, and the contents of the rest of the line as path to the *interpreter*. It invokes the interpreter with the path to the program file as its only argument.

```bash
#!/bin/bash

echo "Hello Hackers!"
```

The script above can be executed as:

```console
hacker@dojo:~$ chmod a+x script.sh
hacker@dojo:~$ ./script.sh
Hello Hackers!
hacker@dojo:~$
```

When `./script.sh` was executed, Linux opened the file, read the first line, extracted `/bin/bash` as the interpreter, and executed `/bin/bash ./script.sh` to launch the script!

Note, the shebang line must be the VERY FIRST line of the file - no blank lines or spaces before it!

There are some common shebangs we might see:

- `#!/bin/bash` for bash scripts
- `#!/usr/bin/python3` for Python scripts
- `#!/bin/sh` for POSIX shell scripts --- this is a more primitive predecessor to `bash` with fewer features, but more compatibility to non-Linux systems!

### Appending arguments to shellscripts

Sometimes, you might need to pass some custom arguments to shellscripts. This makes shellscripts powerful. Consider the following command:

```console
hacker@dojo:~$ bash myscript.sh hello world
```

Inside the script, you can access the passed arguments using special variables:

- `$1` is the first argument ("hello")
- `$2` is the second argument ("world")
- `$3` would contain the third argument (if there is one)

Here is the simple example:

```bash
hacker@dojo:~$ cat myscript.sh
#!/bin/bash
echo "First argument: $1"
echo "Second argument: $2"
hacker@dojo:~$ bash myscript.sh hello world
First argument: hello
Second argument: world
hacker@dojo:~$
```

### Conditionals in bash

Just like with python, you can also use conditionals in bash:

```bash
if [ "$1" == "ping" ]
then
    echo "pong"
fi
```

Bash have weird syntax for some reason, so it can be cryptic compared to other programming languages.

- First, you *must* have spaces after `if`, and after `[` and before `]`.
- Second, `if`, `then`, and `fi` must all be on different lines, or separated by semicolons (`;`); you can't just lump together in a single statement.
- `fi` is the terminator of the `if` statement (`if` backwards). Weird naming but it is what it is and we have to remember it.

The `else` clause executes if the condition is false:

```bash
if [ "$1" == "hello" ]
then
    echo "Hi there!"
else
    echo "I don't understand"
fi
```

`else` doesn't have condition --- it catches everything that didn't match previously. It also doesn't have a `then` statement. Finally, the `fi` goes after the `else` block.

For multiple conditions, you can use `elif`. You do need the `then` after `elif`, just like `if`. As before the `else` at the end catches everything that didn't match.

```bash
if [ "$1" == "one" ]
then
    echo "1"
elif [ "$1" == "two" ]
then
    echo "2"
elif [ "$1" == "three" ]
then
    echo "3"
else
    echo "unknown"
fi
```
