
Did you think you, `hacker`, are the sole ghost in the machine? There are MANY users on a typical Linux system! The full list of users on a Linux system is specified in the `/etc/passwd` file (named so for historical reasons --- it doesn't actually hold passwords anymore). Here is an example from the dojo container:

A lot of users here, and a lot of info! Each line contains, separated by `:`s, the username, an `x` as a placeholder for where the password _used_ to be (we'll cover where it actually is later), the numerical user ID, the numerical default group ID, long-form user details, the home directory, and the default shell.

We can see the `hacker` user at the bottom. That's you! Most of the rest of these users are either there for historical reasons, are service accounts to support various installed software, or some are "utility" accounts (e.g., the `nobody` user is used to ensure that, e.g., some programs run without any special privileges).

One important user is `root`: the system administrator. The system administrator has obvious security implications: a `hacker` user that can somehow, through various functionalities of Linux, become the `root` user would be able to wreak havoc on the system. A very frequent goal of hackers breaking into systems is to escalate to `root`, and thus `root` must be defended at all cost!

### Becoming root with su

Before `sudo`, there is an older and legacy command `su`. While it is typically not used to elevate to `root` access anymore, it was an elegant utility from a more civilized time. `su` is a setuid binary:

```bash
hacker@dojo:~$ ls -l /usr/bin/su
-rwsr-xr-x 1 root root 232416 Dec 1 11:45 /usr/bin/su
hacker@dojo:~$
```

Because it has the SUID bit set, `su` runs as `root`, allowing you to start a `root` shell. To access `su`, you typically need to authenticate with the `root` password:

```bash
hacker@dojo:~$ su
Password: 
su: Authentication failure
hacker@dojo:~$
```

Modern systems rarely have `root` passwords, and different mechanisms (such as `sudo`) are used to grant administrative access.

### Other users with `su`

By default (no provided arguments), `su` will start as root. But you can also use `su` to switch to *that* user instead of `root`. For example:

```bash
hacker@dojo:~$ su some-user
Password:
some-user@dojo:~$
```

### Cracking passwords (JohnTheRipper)

Historically, `/etc/passwd` stores passwords from different users in Linux environment. But since it is a globally-readable file, this is not good for passwords. Hence, passwords are moved at `/etc/shadow`. Here is the example of `/etc/shadow`:


Let's break it down what does the format mean:

- Separated by `:`s, the first and second field of each line respectively indicates username and the password.
- A value of `*` or `!` means that password login for that account is disabled. 
- A blank field means that there is no password (a not-uncommon misconfiguration that allows password-less `su` in some configurations)
- The long string on second line (password) is not the password itself, but the hash of the password. When you input password to the user, it checks the hash value of the inputted password to the hash value of the actual password stored in `/etc/shadow`, if it match, then the password is correct, allowing you to access that user.

But what if you don't know the password? If you have the hashed value of the password, you can *crack* it! `/etc/shadow` is only readable by root by default, but leaks can happen. For example, stored backups can be unencrypted and insufficiently protected on file servers, allowing for countless data disclosures.

If a hacker manages to get their hands on `/etc/shadow`, they can start cracking passwords and wreak havoc. [John the Ripper](https://www.openwall.com/john/) is an infamous cracking tool that can do the job:

```bash
hacker@dojo:~$ john ./my-leaked-shadow-file
Loaded 1 password hash (crypt, generic crypt(3) [?/64])
Will run 32 OpenMP threads
Press 'q' or Ctrl-C to abort, almost any other key for status
password1337      (zardus)
1g 0:00:00:22 3/3 0.04528g/s 10509p/s 10509c/s 10509C/s lykys..lank
Use the "--show" option to display all of the cracked passwords reliably
Session completed
hacker@dojo:~$
```

Here, John the Ripper cracked Zardu's leaked password hash to find the real value of `password1337`. Poor Zardus.

### `sudo` command

`root` passwords are pain to maintain, and their hashes can leak, and is often poorly maintained to well larger environments such as fleets of servers. To address this, the world has moved on from `su` administration to administration via `sudo`. 

`sudo` originally stood for superuser do, but since `su` means "substitute user", `sudo` became "substitute user, do".

Unlike `su`, which defaults to launching a shell as a specified user, `sudo` defaults to running a command as `root`:

```bash
hacker@dojo:~$ whoami
hacker
hacker@dojo:~$ sudo whoami
root
hacker@dojo:~$
```

Or, more relevant to getting flags:

```bash
hacker@dojo:~$ grep hacker /etc/shadow
grep: /etc/shadow: Permission denied
hacker@dojo:~$ sudo grep hacker /etc/shadow
hacker:$6$Xro.e7qB3Q2Jl2sA$j6xffIgWn9xIxWUeFzvwPf.nOH2NTWNJCU5XVkPuONjIC7jL467SR4bXjpVJx4b/bkbl7kyhNquWtkNlulFoy.:19921:0:99999:7:::
hacker@dojo:~$
```

Instead of password authentication, `sudo` checks policies to determine whether the user is authorized to run commands as `root`. These policies are defined in `/etc/sudoers`.
