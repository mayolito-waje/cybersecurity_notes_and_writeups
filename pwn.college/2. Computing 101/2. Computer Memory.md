Registers are **FAST**, but they are *very limited*. Registers are just like **"short term memory"** for humans. When performing specific computation, our brain loads previously learned information into short term memory, acts on that information, then eventually puts new resulting information into our **"long-term memory"**. Humans, in general, can hold and store **5 to 7 pieces of information** in short-term memory, which are very similar to registers in such that specific CPU architecture can only contain limited registers.

**Computer memory** is just like our long-term memory. They are more spacious, but is much **SLOWER** to access. This module will focus on computer memory.

### The need for memory

![Pasted image 20260420081038.png](../../_assets/Pasted%20image%2020260420081038.png)

Registers are *expensive* and are very constrained and limited. We need a place to store lots of data and have *fairly* fast access to it when needed. This is where memory shines.

### Memory: Process perspective

Process memory is used for a lot of use cases: registers, disks, networks, video cards, and so on. Unlike registers, there is too much memory to name every location.

Process memory is addressed linearly.

* From **0x10000** (for security reasons) to **0x7fffffffffff** (for architecture / OS purposes)

Each memory address reference **one byte** in memory. This means 127 *terabytes* of addressable RAM! Just joking, you don't have 127 TB of RAM, they are all ~~fake~~ virtual! Process memory starts out partially filled in by the Operating System.

### Memory (stack)

When a process starts up, a space in memory is created called **stack**. Stack is used for *temporary data storage*. Registers can be **push**ed onto the stack to save values:

````asm
mov rax, 0xc001ca75
push rax
push 0xb0bacafe # WARNING: even on 64-bit x86, you can only push 32-bit immediates...
push rax
````

![Pasted image 20260420090205.png](../../_assets/Pasted%20image%2020260420090205.png)

Values can also be **pop**ped back off the stack (to any register)!

````asm
pop rbx # sets rbx to 0xc001ca75
pop rcx # sets rcx to 0xb0bacafe
````

![Pasted image 20260420090344.png](../../_assets/Pasted%20image%2020260420090344.png)

### Addressing the Stack

The CPU knows where the stack is because its address is stored in **rsp**.

![Pasted image 20260420090753.png](../../_assets/Pasted%20image%2020260420090753.png)

Historical oddity: stack grows backward toward smaller memory addresses! **push** *decreases* rsp, **pop** *increases* it.

### Accessing memory

You can also move data between registers and memory with **mov**.

This will load the 64-bit value stored at memory address **0x12345** into **rbx**:

````asm
mov rax, 0x12345
mov rbx, [rax]
````

This will store the 64-bit value in **rbx** into memory at address **0x133337**:

````asm
mov rax, 0x133337
mov [rax], rbx
````

This is equivalent to push **rcx**:

````asm
sub rsp, 8
mov [rsp], rcx
````

Each addressed memory location contains one byte. An 8-byte write at address **0x133337** will write to addresses **0x133337** through **0x13333f**.

### Controlling write sizes

We can use partials to store/load fewer bits of data. For example, to load 64 bits from address **0x12345** and store the lower 32 bits to address **0x133337**, we can do the following:

````asm
mov rax, 0x12345
mov rbx, [rax]
mov rax, 0x133337
mov [rax], ebx
````

Load 8 bits from address **0x12345** to bh:

````asm
mov rax, 0x12345
mov bh, [rax]
````

Don't forget: changing 32-bit partials (e.g. by loading from memory) zeroes out the whole 64-bit register. Storing 32-bits into memory has no such problems.

### Memory endianess

Data on most modern systems is stored backwards, in *little endian*.

![Pasted image 20260420094344.png](../../_assets/Pasted%20image%2020260420094344.png)

Bytes are *only* shuffled for multi-byte stores and loads of registers to memory. Individual bytes *never* have their bits shuffled. Similarly, writes to the stack behave just like any other write to memory.

**History on little endian**

* Intel created the 8008 for a company called Datapoint in 1972. Datapoint used little endian for simpler implementation of *carry* in arithmetic! For compatibility reasons, Intel also used little endian in 8008 to be used for Datapoint processes. Every step in the evolution between 8008 and modern x86 maintained some level of binary compatibility with its predecessor.

### Address calculation

You can also do some limited calculation for memory addresses.

````asm
mov rax, 0
mov rbx, [rsp+rax*8] # read a qword right at the stack pointer
inc rax
mov rcx, [rsp+rax*8] # read the qword to the right of the previous one
````

You can get the calculated address with Load Effective Address (`lea`).

````asm
mov rax, 1
pop rcx
lea rbx, [rsp+rax*8+5] # rbx now holds the computed address for double-checking
mov rbx, [rbx]
````

### RIP-Relative Addressing

`lea` is one of the few instructions that can directly access the rip register!

````asm
lea rax, [rip] # load the address of the next instruction into rax
lea rax, [rip+8] # the address of the next instruction, plus 8 bytes
````

You can also use `mov` to read directly from those locations.

````asm
mov rax, [rip] # load 8 bytes from the location pointed to by the address of the next instruction
````

Or even write to `rip` register:

````asm
mov [rip], rax # write 8 bytes over the next instruction
````
