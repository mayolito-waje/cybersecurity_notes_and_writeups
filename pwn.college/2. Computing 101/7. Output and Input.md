### Writing Output

Let's say you have an assembly instruction like this:

````asm
mov rdi, 1
mov rsi, [rsp+16]
mov rdx, 1
mov rax, 1
syscall
````

After you compiled it as an executable, you might invoke your program like this:

````text
/tmp/your-program H
````

The output is it writes "H" to `stdout`!

As an explanation, the instruction above *writes one character (the starting character from first program argument) to stdout*. In C-like syntax, you will typically write a statement like this:

````c
write(file_descriptor, memory_address, number_of_characters_to_write)
````

For a concrete example, if you want to write 10 characters starting from some memory address to standard output (*file descriptor 1*), this would be:

````c
write(1, memory_address, 10);
````

Now that is simple! So what is it's similarity to the assembly instruction above? We actually specify the proper parameters in their proper registers!

1. The `rdi` register will serve as the first parameter of the system call. 1 means *file descriptor 1*. Recall that in Linux, FD0 is the Standard *Input*, FD1 is the Standard *Output*, and FD2 is the Standard *Error*. Since we want to write to `stdout` (Standard Output), we will set the `rdi` register to 1.
1. The `rsi` register is the memory address you want to output to `stdout`! In this case, `[rsp+16]` is the memory address of the first argument upon invoking our program, which contain the character 'H'!
1. The `rdx` register holds the number of characters we want to write to standard output. In this case we want to print 1 character to `stdout`.
1. The last one is the syscall index for `write`! In x86 assembly, value 1 is the syscall for `write` so you set 1 to the `rax` register!

### Chaining Syscalls

If you invoke the program above, it will write output to the Standard Output, then will crash! It will probably complains some segfault or something. So how to cleanly exit? After invoking the `write` system call (`mov rax, 1`), we will invoke `exit` syscall to cleanly exit the program.

So how do we invoke two system calls? By invoking two instructions! x86 Assembly allows you to chain multiple syscalls or multiple instructions.

````asm
# write first character of first argument to stdout
mov rdi, 1
mov rsi, [rsp+16]
mov rdx, 1
mov rax, 1
syscall

# now we will invoke exit syscall
mov rax, 60
mov rdi, 42
syscall
````

### Reading Data

Now we know how to output data to console. But how does program receive input data? It `read`s it from stdin!

Just like `write`, `read` is a syscall that shunts data around between file descriptors and memory, and its syscall number is `0`. In C-style syntax, `read` is the same as `write`:

````c
read(0, some_address, 5);
````

This means "read 5 bytes from file descriptor `0` (stdin) into memory starting from `some_address`". So, if you type in (or pipe in) `HELLO HACKERS` into stdin, the above `read` call would result in the following memory configuration:

````text
     Address     │ Contents
+───────────────────────────+
│ some_address   │ 48       │
│ some_address+1 │ 45       │
│ some_address+2 │ 4c       │
│ some_address+3 │ 4c       │
│ some_address+4 │ 4f       │
+───────────────────────────+
````

The numbers contained in memory address are *hexadecimal* representations of ASCII-encoded letters.

Now, putting what our learned into practice, let's read 64 bytes from stdin and print it to stdout!

````
mov rdi, 0
mov rsi, rsp  # stack have plenty of writeable space!
mov rdx, 64
mov rax, 0
syscall

mov rdi, 1
mov rsi, rsp
mov rdx, 64
mov rax, 1
syscall

mov rax, 60
mov rdi, 42
syscall
````

Remember that `rsp` points to the top of the stack, which gives us plenty of room to write so we set the memory address of `rsp` to store our input! The code above, however is **dangerous**. We will learn how to write better assembly while we progress.

### Opening Files

Now that we know how to interact with stdin and stdout, now let's learn how to access files! To access file, we will `open` it using `open` syscall:

````c
open("/flag", 0);
````

The `open` system call (syscall number `2`) takes a pointer to a filename string and returns a brand-new file descriptor referring to that file. The second argument specifies additional modes and permissions for the file. `0` requests the default: read-only.

Registers for `open` follow the same convention!

|Register|Purpose|
|--------|-------|
|`rax`|`2` (syscall number for `open`)|
|`rdi`|pointer to the filename string in memory|
|`rsi`|`0` (read-only)|

When `open` returns, `rax` contains the new file descriptor (fd) number (0 for stdin, 1 for stdout, and 2 for stderr). Other files that are open are just represented by other file descriptors, incrementing from 3 onwards!  We used fd as the first argument to `read`, just like we did for stdin earlier, but this time `read` will read from our file!

### Hardcoding the Filename

If we want to read from filename which is passed as an argument to our program, the instructions might look like this:

````asm
# open and read the file at first argument of program
mov rdi, [rsp+16]
mov rsi, 0
mov rax, 2
syscall

# read from the opened file with the file descriptor being the rax value after open syscall is returned
mov rdi, rax
mov rsi, rsp
mov rdx, 64
mov rax, 0
syscall

# write to stdout, rsi and rdx is already set beforehand from previous syscall
mov rdi, 1
mov rax, 1
syscall

# exit the program cleanly
mov rax, 60
mov rdi, 42
syscall
````

(REMEMBER: **dangerous** code, only for learning purpose)

What, if we already know the path and we want to hardcode it? You can hardcode the filename string directly into your program by writing it onto the stack, byte by byte! The `open` syscall needs a *pointer* to the filename, so you need the bytes `/ f l a g` stored somewhere in memory. You already know a writable memory address: `rsp` (the stack). You can write each character one byte at a time:

````asm
mov BYTE PTR [rsp], '/'
mov BYTE PTR [rsp+1], 'f'
mov BYTE PTR [rsp+2], 'l'
mov BYTE PTR [rsp+3], 'a'
mov BYTE PTR [rsp+4], 'g'
mov BYTE PTR [rsp+5], 0
````

A few things to note here:

* **`BYTE PTR`**: When you write to a memory address like `[rsp]` using an *immediate value* (a number or character), the CPU doesn't know how many bytes you intend to write --- one? two? eight? `BYTE PTR` is a *size directive* that tells the assembler "I mean exactly one byte." Without it, the assembler won't know what you want and will refuse to assemble the instruction.

* **Single quotes**: In assembly, a single-quoted character like `'f'` represents that character's one-byte ASCII value. So `'f'` is just a convenient way of writing `0x66`, and `'/'` is `0x2f`.

* **The null byte**: The last byte we write is `0` --- a special *null* byte. This is how Linux knows where a string ends: it reads bytes starting from the pointer you give it and stops when it hits a `0` byte. Without it, `open` would keep reading past `"flag"` into whatever else is on the stack, and you'd be trying to open a file with a nonsense name!

After writing these bytes, `rsp` points to the null-terminated string `"/flag"`, ready to pass to `open`.
