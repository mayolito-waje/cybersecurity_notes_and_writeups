### `show ip interface brief`

![Pasted image 20260502141049.png](../_assets/Pasted%20image%2020260502141049.png)

Using `show ip interface brief` on SW1 using Cisco CLI, notice the four connected interfaces `Status` and `Protocol`. We haven't configured anything on the Cisco router except configuring the `hostname`, yet the connected interface's status and protocol are all `up`.  This is the difference between Cisco router and a Cisco switch! Unlike router, the status (layer 1) and protocol (layer 2) state of the connected interfaces in a Cisco switch don't have `shutdown` applied by default. The `IP-Address` are `unassigned` because this is a layer 2 Switch! Switch don't need IP addresses, unless it is a multilayer switch (which we will learn in the future lessons).

![Pasted image 20260502141729.png](../_assets/Pasted%20image%2020260502141729.png)

Now let's focus on columns `Status` and `Protocol` on interfaces `FastEthernet0/5` and up. They are fundamentally different from Cisco router's `administratively down` Status because `down/down` in a Cisco switch simply means there are no connected devices in that interface!

* **Router** interfaces have the `shutdown` command applied by default, and will be in the `administratively down/down` state by default
* **Switch** interfaces do NOT have the `shutdown` command applied by default, and will be in the `up/up` state if connected to another device OR in the `down/down` state if not connected to another device. No manual configuration in the CLI needed!

### `show interfaces status`

![Pasted image 20260502142448.png](../_assets/Pasted%20image%2020260502142448.png)

`show interfaces status` is another useful command to look for status of interfaces on a Switch. Now let's look what those fields means:

* **Port** - simply lists interfaces
* **Name** - similar to a description field from `show interfaces description` command in Cisco routers.
* **Status** - status of the interfaces, the most common being `connected` and `notconnect`. We will learn about other statuses later.
* **Vlan** - Vlan divide LANs into smaller LANs. Default VLAN of 1. Notice that `Fa0/2` Vlan is `trunk`. We will learn more about this later.
* **Duplex** - as we learned in previous lessons, full duplex is capable of transmitting and receiving data to/from a device. If it is not, it is called *half duplex*. By default, `Duplex` is `auto` in Cisco switches, meaning it will negotiate in a neighboring device, and will use full duplex if possible. That's why interfaces not connected have a value of `auto`, and connected interfaces have value of `a-full` (auto full).
* **Speed** - also `auto` by default, meaning they are able to negotiate and use the fastes speed the connected device is capable of. We are using FastEthernet, so they are capable of having a speed of 100 megabits per second, so the value is `a-100` (100 mbps is auto negotiated with a neighboring device).
* **Type** - The type of interface being used to connect.

### Full/Half Duplex

* **Half duplex** - the device cannot send and receive data at the same time. If it is receiving the frame, it must wait before sending the frame.
* **Full duplex** - the device **can** send and receive data at the same time. It does not have to wait.

Modern networks that use switches almost always use full-duplex because it is simply better.

### LAN Hubs

![493](../_assets/Pasted%20image%2020260502150513.png)

Where is half duplex used nowadays? The answer is *almost nowhere*. But before switches we have *Hubs* that use these half duplex connection. Hubs are way simpler than switches, in fact it is a repeater. Any frame it receives, it always *flood* it with a broadcast or unknown unicast frame.

Let's say PC1 want to send to PC2 under a Hub connection:

![391](../_assets/Pasted%20image%2020260502151128.png)
![393](../_assets/Pasted%20image%2020260502151156.png)
![394](../_assets/Pasted%20image%2020260502151226.png)

Neat, it works if there are no traffic. But what if PC1 wants to send a message to PC2 and PC3 wants to send a message to PC1? Remember that Hub uses *half-duplex* connection. So, let's simulate it:

![407](../_assets/Pasted%20image%2020260502151414.png)![408](../_assets/Pasted%20image%2020260502151456.png)
![410](../_assets/Pasted%20image%2020260502164059.png)

This is where the problem occurs, the hub won't send the first frame first and send the another one after! It will simply tries to *flood both* at the same time.

![414](../_assets/Pasted%20image%2020260502151739.png)

PC2 won't receive either frame intact due to collision on the interface!

![Pasted image 20260502151912.png](../_assets/Pasted%20image%2020260502151912.png)

All devices connected to a Hub are part of what's called a **Collision Domain**. Because such collisions can occur due to limitation of half-duplex connection.

### CSMA/CD

**CSMA** - Carrier Sense Multiple Access with Collision Detection

* CSMA is a mechanism to prevent such collisions in half-duplex connections.
* Before sending frames, devices 'listen' to the collision domain until they detect that other devices are not sending.
* If a collision does occur, the device sends a jamming signal to inform the other devices that a collision happened.
* Each device will wait a random period of time before sending frames again.
* The process repeats.

### Collision Domains

Hubs with CSMA/CD was the norm to prevent such collisions back then. Hubs are simple repeaters that operate at *Layer 1*, repeating whatever signals they receives. Hubs are now replaced by switches, that operates at *Layer 2*, that uses MAC addresses to send frames to specific hosts.

![Pasted image 20260502153828.png](../_assets/Pasted%20image%2020260502153828.png)

With switches, you effectively have 3 collision domains, one for each devices. So devices do not have to worry if other devices sent some network traffic. These allows devices to utilize full-duplex connection without worry. Collisions do happen sometimes, but it is very rare occurrence and is mostly due to some misconfiguration rather than regular occurrence.

### Full/Half Duplex Review

* **Half duplex:** The device cannot send and receive data at the same time. If it is receiving a frame, it must wait before sending a frame. Devices attached to a **hub** must operate in half duplex.
* **Full duplex:** The device can send and receive data at the same time. It does not have to wait. Devices attached to a  **switch** can operate in full duplex.

### Speed/Duplex Autonegotiation

* Interfaces that can run at different speeds (10/100 or 10/100/1000) have default settings of **speed auto** and **duplex auto**
* Interfaces 'advertise' their capabilities to the neighboring device, and they negotiate the best **speed** and **duplex** settings they are both capable of.

![Pasted image 20260502154831.png](../_assets/Pasted%20image%2020260502154831.png)
![Pasted image 20260502154930.png](../_assets/Pasted%20image%2020260502154930.png)

![Pasted image 20260502155201.png](../_assets/Pasted%20image%2020260502155201.png)
![Pasted image 20260502155354.png](../_assets/Pasted%20image%2020260502155354.png)
![Pasted image 20260502155510.png](../_assets/Pasted%20image%2020260502155510.png)

Since autonegotiation is disabled in PC3 (the blue one) and uses 100 Mbps speed, the switch with autonegotiation enabled will sense its speed and use 100 Mbps connection. However, since it is 100 Mbps, the switch will set the duplex to **half-duplex** on that interface when PC3 actually uses *full-duplex*. This will lead to **duplex mismatch**, where collisions would occur resulting in poor performance. That's why it is better to enable autonegotiation if it is not enabled by default!

### Interfaces Errors

Let's look some of the errors that can occur on interfaces that otherwise seem to be working. We can also use `show interfaces` on Cisco switches:

![Pasted image 20260502160041.png](../_assets/Pasted%20image%2020260502160041.png)

![565](../_assets/Pasted%20image%2020260502160908.png)

* `269 packets input, 71509 bytes` pertains to total number of packets that interface received, and the total number of bytes in those packets.
* **Runts:** frames that are smaller than the minimum frame size (64 bytes)
* **Giants:** frames that are larger than the maximum frame size (1518 bytes)
* **CRC:** frames that failed the CRC check (in the Ethernet FCS trailer)
* **Frame:** frames that have an incorrect format (due to error)
* **Input errors:** total of various counters, such as the above four
* **Output errors:** frames the switch tried to send, but failed due to an error
