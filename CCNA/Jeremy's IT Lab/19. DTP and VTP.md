* DTP and VTP are Cisco proprietary and is only available in Cisco devices
* DTP and VTP were removed form the CCNA exam topics list for the new exam (200-301). However, it's important to know their function, and you may still get questions about them on the exam even though they are not on the topics list.
* DTP = Dynamic Trunking Protocol
* VTP = VLAN Trunking Protocol

### Dynamic Trunking Protocol (DTP)

* DTP is a Cisco proprietary protocol that allows Cisco switches to dynamically determine their interface status (**access** or **trunk**) without manual configuration.
* DTP is enabled by default on all Cisco switch interfaces.
* We don't need to use either `switchport mode access` or `switchport mode trunk` if DTP is enabled.
* For security purposes, manual configuration is recommended. DTP should be disabled on all switchports.

![ 500](../_assets/Pasted%20image%2020260525143943.png)

* `switchport mode dynamic` sets the interface to DTP mode.

![ 500](../_assets/Pasted%20image%2020260525144156.png)

### DTP Mode: `dynamic desirable`

A switchport in **dynamic desirable** mode will actively try to form a trunk with other Cisco switches. It will forma  trunk if connected to another switchport in the following modes:

* `switchport mode trunk`
* `switchport mode dynamic desirable`
* `switchport mode dynamic auto`

**Examples:**

![ 500](../_assets/Pasted%20image%2020260525144657.png)
![ 500](../_assets/Pasted%20image%2020260525144714.png)
![ 500](../_assets/Pasted%20image%2020260525144732.png)

Because SW2 is a manually configured trunk port, SW1 automatically sets its connected interface to SW2 in trunk mode, thanks to its `dynamic desirable` administrative mode.

![ 500](../_assets/Pasted%20image%2020260525144919.png)

If SW2's administrative mode becomes `dynamic desirable`, Both SW1 and SW2 will agree that their connected interfaces be set to trunk port because of DTP.

![ 500](../_assets/Pasted%20image%2020260525145128.png)

If SW2 is set to `switchport mode dynamic auto`, it won't actively try to form a trunk. It is more passive. If SW1 want to form a trunk, SW2 will form a trunk but it won't actively try to form a trunk with SW2 (that's the best explanation I can give). However, because SW2 is set with `switchport mode dynamic desirable` mode, SW1 and SW2 will form a trunk based on `dynamic desirable` rules.

![ 500](../_assets/Pasted%20image%2020260525145630.png)

However, if SW2's interface is set to access port, `dynamic desirable` will set it's interface to access mode because it will only actively set the interface to trunk port if the other switch is in `trunk`, `dynamic desirable`, or `dynamic auto` mode.

* As you can see in `Operational Mode`, both switches interfaces are set to `static access`.
* `static access` means an access port that belongs to a single VLAN that doesn't change (unless you configure a different VLAN).
* There are also `dynamic access` ports, in which a server automatically assigns the VLAN depending on the MAC address of the connected device (**this is out of the scope of the CCNA**)

### DTP Mode: `dynamic auto`

A switchport in `dynamic auto` mode will NOT actively try to form a trunk with other Cisco switches,s however it will form a trunk if the switch connected to it is actively tyring to form a trunk. It will form a trunk with a switchport in the following modes:

* `switchport mode trunk`
* `switchport mode dynamic desirable`

**Examples:**

![ 500](../_assets/Pasted%20image%2020260525150358.png)

If SW1 is in `dynamic auto` mode and SW2 is `trunk` mode, Both switches will agree to form a trunk port.

![ 500](../_assets/Pasted%20image%2020260525150551.png)

If both SW1 and SW2 are in `switchport mode dynamic auto` mode, both will not actively try to form a trunk, so they will agree to set their connected interfaces as access port.

![ 500](../_assets/Pasted%20image%2020260525150704.png)

As you can guess, if SW2 interface is an access port, SW1's interface will also set itself as an access port because of `dynamic auto` mode.

### Disabling DTP

* On older switches, `switchport mode dynamic desirable` is the default administrative mode.
* On newer switches, `switchport mode dynamic auto` is the default administrative mode.
* You can disable DTP negotiation on an interface with this command: `switchport nonegotiate`
* Configuring an access port with `switchport mode access` also disables DTP negotiation on an interface.
* It is recommended that you disable DTP on all switchports and manually configure them as access or trunk ports.

### Trunk Encapsulation Negotiation via DTP

* Switches that support both **802.1Q** and **ISL** trunk encapsulations can use DTP to negotiate the encapsulation they will use.
* This negotiation is enabled by default, as the default trunk encapsulation mode is: `switchport trunk encapsulation negotiate`
* **ISL** is favored over **802.1Q**, so if both switches support ISL it will be selected.
* The DTP frames that DTP uses are sent in VLAN1 when using **ISL**, or in the native VLAN when using **802.1Q** (the default native VLAN is VLAN1, however).

To show how DTP negotiation works, let's look at the following configuration using `show interfaces <interface> switchport`

![ 500](../_assets/Pasted%20image%2020260525153334.png)

Both SW1 and SW2 are set to `switchport mode dynamic desirable` so they would form a trunk.

![ 500](../_assets/Pasted%20image%2020260525153522.png)

Notice that the default trunking encapsulation of `negotiate` results in operation trunking encapsulation of `isl`.

![ 500](../_assets/Pasted%20image%2020260525153709.png)

`Negotiation of Trunking` shows whether DTP is enabled, whether the interfaces are sending DTP frames or not. If the interfaces are in `dynamic desirable`, `dynamic auto`, or `trunk` mode, it will be `On`. If the interface is in `access` mode or you used `switchport nonegotiate`, it will be `Off`.

### VTP (VLAN Trunking Protocol)

* VTP allows you to configure VLANs on a central VTP server switch, and other switches (VTP clients) will synchronize their VLAN database to the server.
* It is designed for large networks with many VLANs, so that you don't have to configure each VLAN on every switch.
* Like DTP, it it rarely used, and it is recommended that you do not use it.
* There are three VTP versions: 1, 2, and 3.
* There are three VTP modes that switches can operate in: **server**, **client**, and **transparent**.
* Cisco switches operate in VTP server mode by default.

### VTP Servers/Clients

**VTP Servers**:

* Can add/modify/delete VLANs.
* Store the VLAN database in non-volatile RAM (NVRAM). This means that the VLAN database is saved even when the switch is turned off or reloaded.
* Will increase the **revision number** every time a VLAN is added/modified/deleted. The revision number is very important part of VTP. It's what VTP uses to determine the *newest version* of the VLAN database and the version that the switches will synchronize to.
* Will advertise the latest version of the VLAN database on **trunk** interfaces, and the VTP clients will synchronize their VLAN database on it.
* **VTP servers also function as VTP clients. Therefore, a VTP server will synchronize to another VTP server with a higher revision number.**

**VTP Clients:**

* Cannot add/modify/delete VLANs.
* Do not store the VLAN database in NVRAM. (**in VTPv3, they do**)
* Will synchronize their VLAN database to the server with the highest revision number in their VTP domain.
* Will advertise their VLAN database, and forward VTP advertisements to other clients over their trunk ports.

### The danger of VTP

If you connect an old switch **with a higher revision number** to your network (and the VTP domain matches), all switches in the domain will sync their VLAN database to that switch!

This could cause all of the hosts on your network to instantly lose connectivity because the switches could sync to a totally different VLAN database and the VLANs the network had been using could've disappear. This is why VTP is not really used in modern networks.

![ 500](../_assets/Pasted%20image%2020260525165601.png)

Let's say that this is your current network where SW1, SW2, and SW3 share the same VTP domain and VLAN database.

![ 500](../_assets/Pasted%20image%2020260525165723.png)

Let's say you connected an old switch with same VTP domain name but with **higher revision**.

![ 500](../_assets/Pasted%20image%2020260525165812.png)

SW4 will send VTP advertisements with revision number `50` and forward it throughout the domain.

![ 500](../_assets/Pasted%20image%2020260525165940.png)

All switches will update their VLAN database to match. All of the hosts in VLAN 10, 20, 30, and 40 will suddenly lose connectivity!

### VTP Transparent Mode

**VTP Transparent Mode:**

* Does not participate in the VTP domain (does not sync its VLAN database).
* Maintains its own VLAN database in NVRAM. It can add/modify/delete VLANs, but they won't be advertised to other switches.
* Although it doesn't sync its own VLAN database, it will forward VTP advertisements over its trunk port if the VTP advertisement is in the same domain as it.
