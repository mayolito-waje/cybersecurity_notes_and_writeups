### Ethernet Frame

![524](../../_assets/Pasted%20image%2020260429061146.png)

The **Preamble + SFD** is usually *not* considered part of the Ethernet header. Therefore, the size of the Ethernet header + trailer is 18 bytes (6 + 6 + 2 + 4). The minimum size for an Ethernet frame (header + payload \[packet\] + trailer) is **64 bytes**. Therefore, the minimum payload (packet) size should be 46 bytes (64 bytes - 18 bytes \[header + trailer\] = 46).

What if you send a smaller packet such as 34 bytes? Well, it will still send it as 46 bytes! If the payload is *less than 46 bytes*, padding bytes will be added (populates the packet with extra zero's). For example, a 34 byte packet will have 12 byte padding to comply to 48 byte minimum rule!

### Address Resolution Protocol (ARP)

Enter ARP! As mentioned with the previous problem above, we need to know the MAC address of PC3 to send message to. ARP handles it. ARP is used to discover the Layer 2 address (MAC address) of a known Layer 3 address (IP address).

ARP consists of two messages: **ARP Request** and **ARP Reply**. PC1 sends ARP Requests, and PC3 will send ARP Reply. The **ARP Request** is the **broadcast**, which send it to all hosts on the network. Once PC3 receives the request, it will send an **ARP Reply** which is the **unicast** (ARP Reply sent only to one host which is the host that sent the request).

### Ping

`ping` is a network utility that is used to test reachability, for example, to test if 2 computers can reach each other. It measure's **round-trip time**. For example, the time from PC1 to PC3, back to PC1.

Ping have two messages: **ICMP Echo Request** and **ICMP Echo Reply**.

**ICMP Echo Request** - Unlike ARP, the sender won't broadcast the ICMP Echo Request. It will be sent to a specific host. It *has* to know the address of the destination host, which is why ARP must be used first. To use ICMP Echo Request, you usually use a command like `ping <ip-address>` in common operating systems.

![Pasted image 20260429153351.png](../../_assets/Pasted%20image%2020260429153351.png)

Notice that 1 ping failed. It is because PC1 doesn't know where `192.168.1.3` is at first `ping`. So it has to use ARP to know where it is.

![Pasted image 20260429153903.png](../../_assets/Pasted%20image%2020260429153903.png)

If you look at cisco's privileged EXEC mode using `show arp` command, noticed that `192.168.1.3` is dynamically learned which is the PC3!

![Pasted image 20260429154046.png](../../_assets/Pasted%20image%2020260429154046.png)

Now let's look deeper. We captured the packet stream using Wireshark! Notice #2 and #3. #2 is the ARP request, where the Source is the PC1.

* PC1 with the address of `0c:2f:b0:11:9d:00` sends an ARP request to the broadcast address. Notice the request info: `Who has 192.168.1.3? Tell 192.168.1.1`.
* PC3 with the address of `0c:2f:b0:6a:39:00` received the ARP request and responded with `192.168.1.3 is at 0c:2f:b0:6a:39:00`. It then sent an ARP reply to `0c:2f:b0:11:9d:00` which is the MAC of PC1!
* \#4 to #11 is the successful stream of ICMP Echo request and ICMP echo reply because PC1 and PC3 already know each other's MAC addresses!
* Notice that each ICMP Echo Request has source IP address of `192.168.1.1` (PC1 IP) and each ICMP Echo Reply has source IP address of `192.168.1.3` (PC3 IP)

### Clearing the MAC Address Table

We learned before that dynamically learned MAC addresses are removed from the table after around ~5 minutes. This is also known as **Aging**. We can, however, manually delete dynamic MAC address using Cisco CLI with the command `clear mac address-table dynamic`.

![Pasted image 20260429155722.png](../../_assets/Pasted%20image%2020260429155722.png)

If you just want to remove specific dynamic MAC address, you can use this command: `clear mac address-table dynamic address <MAC-address>`.

![Pasted image 20260429155856.png](../../_assets/Pasted%20image%2020260429155856.png)

You can see in the output above that the entry for PC1's MAC address in the MAC address table has been removed.

To delete dynamic mac addresses under some specified port, you can use this command: \`clear mac address-table dynamic interface <interface-id>.

### Ethernet Frame review

![Pasted image 20260429160537.png](../../_assets/Pasted%20image%2020260429160537.png)

What if you ping an IP with a specified size (e.g. using `ping <ip> size 36`)? Notice the **Padding**. The minimum payload size for an Ethernet payload should be **46 bytes**. However we *only* sent 36 bytes. So there should be an *extra padding* of **10 bytes**. In the output above, the 0's in Padding information are in *hexadecimal*. Each digit in hexadecimal is 4 bits, so 2 digits equals 8 bits which is equal to 1 byte. There are 20 0's in the output above. So 20/2 is 10, which means *10 bytes*.

![Pasted image 20260429161412.png](../../_assets/Pasted%20image%2020260429161412.png)

If you zoom out of Wireshark, you can actually see the hexadecimal equivalent for the payload! Notice that in the end, 10 bytes of padding is added!
