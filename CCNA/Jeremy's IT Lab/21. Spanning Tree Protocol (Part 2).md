### STP Port States

![ 500](../_assets/Pasted%20image%2020260531081436.png)

* Root/Designated ports remain stable in **Forwarding** state.
* Non-designated ports remain stable in a **Blocking** state.
* **Listening** and **Learning** are transitional states which are passed through when an interface is activated, or when a **Blocking** port must transition to a Forwarding state due to a change in the network topology.
* There is a also a **Disabled** state which just simply mean that the interface is administratively down or disabled.

---

### STP Port States - Blocking

* Non-designated ports are in a **Blocking** state
* Interfaces in a Blocking state are effectively disabled to prevent loops.
* Interfaces in a Blocking state do not send/receive regular network traffic.
* Interfaces in a Blocking state **receive** STP BPDUs (so the interface can prepare from transition from blocking state to forwarding state if topology changes).
* Interfaces in a Blocking state do NOT forward STP BPDUs.
* Interfaces in a Blocking state do NOT learn MAC addresses (don't put the MAC address in MAC address table).

### STP Port States - Learning

* After the Blocking state, interfaces with the Designated or Root role enter the **Listening** state.
* Only **Designated** or **Root** ports enter the Listening state (Non-designated ports are always Blocking).
* The Listening state is 15 seconds long by default. this is determined by the **Forward delay** timer.
* An interface in the Listening state ONLY forwards/receives STP BPDUs.
* An interface in the Listening state does NOT send/receive regular traffic (including broadcasts)
* An interface in the Listening state does NOT learn MAC addresses from regular traffic that arrives on the interface.

### STP Port States - Learning

* After the Listening state, a Designated or Root port will enter the **Learning** state.
* The Learning state is 15 seconds long by default. This is determined by the **Forward delay** timer (the same timer is used for both the Listening and Learning states)
* An interface in the Learning state ONLY sends/receives STP BPDUs.
* An interface in the Learning state does NOT send/receive regular traffic.
* An interface in the Learning state **learns** MAC address from regular traffic that arrives on the interface.

### STP Port States - Forwarding

* Root and Designated ports are in a **Forwarding** state.
* A port in the Forwarding state operate as normal.
* A port in the Forwarding state sends/receives BPDUs.
* A port in the Forwarding state sends/receives normal traffic.
* A port in the Forwarding state learns MAC addresses.

### STP Timers - Hello

![ 500](../_assets/Pasted%20image%2020260531083735.png)

Let's assume all the switches all come online at the same time.

![ 500](../_assets/Pasted%20image%2020260531083837.png)

All the switches will assume they are the Root Bridge, so they will initially send Hello BPDUs.

![ 500](../_assets/Pasted%20image%2020260531083925.png)

Once the the designated and non-designated ports are stabilized, only the Root Bridge will send the BPDUs.

![ 500](../_assets/Pasted%20image%2020260531084015.png)

The receiving switches will then forward the BPDUs on their **designated** ports.

![ 500](../_assets/Pasted%20image%2020260531084303.png)

Fast forward 2 seconds later, the Root Bridge will send the BPDUs again.

![ 500](../_assets/Pasted%20image%2020260531084403.png)

The receiving switches will then again forward the BPDUS on their **designated** ports.

**NOTE:** Switches do not forward the BPDUs out of their **root** ports and **non-designated** ports, only their **designated** ports.

---

### STP Timers - Forward Delay

Forward Delay is used to determine how long the switch will in the Listening and Learning states (each state is **15** seconds = 30 seconds).

---

### STP Timers - Max Age

Max Ages is used to determine how long an interface will wait to change the STP topology **after ceasing to receive Hello BPDUs**. The timer is reset every time a BPDU is received. The default duration is 20 seconds (10 seconds * Hello BPDUs)

![ 500](../_assets/Pasted%20image%2020260531085424.png)

Remember that designated ports will forward received BPDUs. All root ports and non-designated ports will receive BPDUs.

![ 500](../_assets/Pasted%20image%2020260531085608.png)

Let's focus on SW2's `G0/1` interface. After the SW2 received the BPDU on interface `G0/1`, it will start counting for 20 seconds until it receives another BPDUs again.

![ 500](../_assets/Pasted%20image%2020260531085814.png)

After 2 seconds, the Root Bridge will send BPDUs again. After SW2 receives it on `G0/1` interface, it will reset the timer again.

![ 500](../_assets/Pasted%20image%2020260531085906.png)

What if failure occurs between SW1 and SW2?

![ 500](../_assets/Pasted%20image%2020260531090004.png)

The Root Bridge will keep sending BPDUs every 2 seconds, but SW2's `G0/1` interface will not receive it. So the Max Age Time will keep counting down until it reaches `0`.

So what is **Max Age** STP Timer?

* If another BPDU is received before the max age timer counts down to 0, the time will reset to 20 seconds and no changes will occur.
* If another BPDU is not received, the max age timer counts down to 0 and the switch will reevaluate its STP choices, including root bridge, and local root, designated, and non-designated ports.
* If a non-designated port is selected to become a designated or root port, it will transition from the blocking state to the listening state (15 seconds), learning state (15 seconds), and then finally the forwarding state. So, it can take a total of **50 seconds** for a blocking interface to transition to forwarding.
* The 50 seconds duration is deliberate because these timers and transitional states are to make sure that loops aren't accidentally created by an interface moving to forwarding state too soon.

**NOTE:**

* A forwarding interface can move directly to a blocking state (there is no worry about creating a loop by blocking in interface).
* A blocking interface cannot move directly to forwarding state. It must go through the listening and learning states.

### Spanning Tree BPDU (Wireshark)

![ 500](../_assets/Pasted%20image%2020260531152325.png)

#### Ethernet II

* In Ethernet header section, remember the destination, which uses PVST+ and destination MAC of `01:00:0c:cc:cc:cd`. PVST is an older version that supports only ISL trunk encapsulation. PVST+ supports 802.1Q encapsulation.
* Regular STP (not Cisco's PVST+) uses a destination MAC address of `01:80:c2:00:00:00`.

#### Spanning Tree Protocol

* The `Protocol Identifier` uses hex `0x0000` for STP
* The `Protocol Version Identifier` uses `0` which is for classic spanning tree (rapid spanning trees have different version).
* The `BPDU Type` is a configuration BPDU which uses hex of `0x00`.

#### BPDU flags

* Use to signal topology changes. Not that important in CCNA.

#### Root Identifier

* Identifies the Bridge ID of the Root Bridge.
* The `Root Bridge Priority` is the BID, the `Root Bridge System ID` is the extended system ID, the `Root Bridge System ID` is the MAC address of the root bridge.
* We set the MAC address of switch with `aa:aa:aa:aa:aa:aa` and set it as the root bridge.

#### Root Path Cost

* Since this packet comes from the root bridge, the root path cost is 0.

#### Bridge Identifier

* Since the packet comes from root bridge, we can see that the values of `Root Identifier` and the `Bridge Identifier` is the same.

#### Port identifier

* The interface which sent the BPDU.
* The value of `0x8002` means that it uses default port priority (which has the value of `128`, or `80` in hex, which is the first value in the mentioned hex). The latter half of hex, `02`, means the number of the ports itself.

#### Timers (Message Age, Max Age, Hello Time, Forward Delay)

* `Message Age` starts at `0` at the Root Bridge, and increases by 1 each time it is forwarded by another switch. It is subtracted from the `Max Age` when the switch receives the BPDU. If the BPDU is passed through 5 switches, the 6th bridge will immediately reduce its `Max Age` timer to `15`, so each time the bridge receives a BPDU, the `Max Age` will reset to `15` instead of `0` even though the `Max Age` timer is `20`.
* The `Max Age`, `Hello Time`, and `Forward Delay` are the topics we talked about.
* Remember, **the STP timers on the root bridge determine the STP timers for the entire network** even if they are configured differently.

### STP Toolkit - Portfast

![Pasted image 20260531160209.png](../_assets/Pasted%20image%2020260531160209.png)

* If you connect a switch to an end host or other switches, the interface will have an **orange** indicator, or orange blinking lights on fresh connection. This means that it is still in **Listening** or **Learning** state and won't receive/forward traffic yet.
* As we have learned before, it is a feature because Layer 2 loops are dangerous so it is necessary to enter Listening and Learning state first before the interfaces can forward traffic.

![Pasted image 20260531160532.png](../_assets/Pasted%20image%2020260531160532.png)

* Once you see the green lights, it means it is now in forwarding state.
* This will typically take 30 seconds because it takes 15 seconds to be in Listening mode and another 30 seconds to be in Learning mode.

![ 500](../_assets/Pasted%20image%2020260531155740.png)

* You should only use Portfast if the interface is connected to endhosts.
* Portfast bypasses the **Listening** and **Learning** state on connected interfaces.
* While you can configure Portfast on a trunk port, it won't have any effect so it only works on access ports (which is the switchport you use when connecting to end hosts).

![ 500](../_assets/Pasted%20image%2020260531155827.png)

* Use `spanning-tree portfast` to set the interface in Portfast mode. Remember the `%Warning:` caution! Only use Portfast if you connect to end hosts and not on other switches!
* You can also use the command `spanning-tree portfast default` while in global config mode to enable portfast on all **access ports** (not trunk ports).

Portfast can still pose a risk.

![ 500](../_assets/Pasted%20image%2020260531161002.png)

If SW2's `G0/2` and SW3's `G0/2` interface are in Portfast mode which was originally connected to an end host, what if an employee unknowingly replaced the PC with another switch? It will cause a layer 2 loop! So always be careful when setting up Portfast.

### STP Toolkit - BPDU Guard

If an interface with BPDU Guard enabled receives a BPDU from another switch, the interface will be shut down to prevent a loop from forming.

![ 500](../_assets/Pasted%20image%2020260531161630.png)

From interface configuration mode, use the command `spanning-tree bpduguard enable`. Similar to Portfast, there is an also an option to enable it by default. While in global config mode, the command `spanning-tree portfast bpduguard default` enables BPDU Guard on all Portfast-enabled interfaces.

![ 500](../_assets/Pasted%20image%2020260531161904.png)

If you connect a switch to a BPDU Guard enabled interface, you can see in the output above what happens when a BPDU arrives on a BPDU Guard enabled port. The port is disabled, and is effectively shut down.

![ 500](../_assets/Pasted%20image%2020260531162205.png)

If you want to enable the port again, we can simply `shutdown` then `no shutdown` the interface. You can see that the interface comes up. But if you didn't actually solve the problem and it is still connected to a switch, you can see that the interface will shutdown again when the next BPDU arrives. So make sure to actually solve the problem before trying to enable the interface again!

---

### STP Toolkit - Root Guard/Loop guard

* **Root Guard:** If you enable **root guard** on an interface, even if it receives a superior BPDU (lower bridge ID) on that interface, the switch will not accept the new switch as the root bridge. The interface will be disabled. This helps maintain the STP if someone plugs another switch in another network either with bad intent or without knowing the impact of their action.
* **Loop Guard:** If you enabled **loop guard** on an interface, even if the interface stops receiving BPDUs, it will not start forwarding. The interface will be disabled. This is good for cases when the switch can't receive data but can forward it or vice versa.

**NOTE:** You probably don't have to know these STP optional features (or others such as UplinkFast, Backbone Fast, etc) for the CCNA. But make sure you know **Portfast** and **BPDU Guard**. If you want to read more about the others just in case, do a Google search.

### STP Configuration - Mode

![ 500](../_assets/Pasted%20image%2020260531184023.png)

In global config mode, you can set up the spanning tree mode using the command `spanning-tree mode <mode>`

* `mst` - Multiple Spanning Tree mode, not covered in CCNA.
* `pvst` - Per-VLAN Spanning Tree mode, classic STP with Cisco proprietary where there is different spanning tree per VLAN.
* `rapid-pvst` - Per-VLAN Rapid Spanning Tree mode, we will learn this later.

Modern Cisco switches uses Rapid PVST mode by default, and there usually no reason to change it. But if we want to use classic spanning tree, we can use the command `spanning-tree mode pvst`.

### STP Configuration - Root Bridge

![ 500](../_assets/Pasted%20image%2020260531184600.png)

By default, the topology above will use SW1 as the root bridge. But you can also manually configure the root bridge. What if we want to configure SW3 as the root bridge? We could also configure a secondary root bridge! If the current root bridge fails, the secondary root bridge can come to the rescue next in line.

![ 500](../_assets/Pasted%20image%2020260531184852.png)

We can use the command `spanning-tree vlan <vlan-number> root primary`. It sets the STP priority of the switch to `24576`. If another switch already has a priority lower than `24576`, it sets this switch's priority to `4096` less than the other switch's priority.

If we check the running configuration of the switch, we can see that 3 commands are applied with the command `spanning-tree vlan <vlan-number> root primary`:

![ 500](../_assets/Pasted%20image%2020260531185402.png)

The `spanning-tree vlan <vlan-number> root secondary` command sets the STP priority to `28672`. Just like with the previous command, the running config will also set 3 commands:

### STP Load-Balancing

![ 500](../_assets/Pasted%20image%2020260531190036.png)

Using PVST+, the above is the topology for VLAN1. You can see that the interface connection between SW1 and SW2 is blocked.

Now what about VLAN2? The default topology will look like this:

![ 500](../_assets/Pasted%20image%2020260531190155.png)

The topology is almost identical, except the connection between SW1 and SW2 won't be disabled. Instead, the connection between SW2 and SW3 will be disabled. Why? This allows for what's called **spanning tree load-balancing**.

* If you have multiple VLANs in your network, blocking the same interface in each VLAN, is a waste of interface bandwidth.
* That connection will be doing nothing, just waiting for another connection to fail, so it can start forwarding.
* However, if you configured different root bridge for different VLANs, different VLANs will disable different interfaces.

**SW1 (in global config mode)**:

1. `spanning-tree vlan 20 root secondary`

**SW2 (in global config mode):**

1. `spanning-tree vlan 20 root primary`
1. `spanning-tree vlan 10 root secondary`

With the above settings, perhaps the VLAN 10 topology would look like this:
