### Local Area Networks (LANs)

![Pasted image 20260428080541.png](../../_assets/Pasted%20image%2020260428080541.png)

In simple concept, a Local Area Network (LAN) is a relatively contained local network access over a small area such as office floor or home network. Routers are used to connect separate LANs. By the definition mentioned, the diagram above have 4 LANs. A single LAN can have more than 1 switch. Switches do not separate LANs, but can be used as an extension to existing LAN as long as switches are connected to each other.

### Ethernet Frame

![445](../../_assets/Pasted%20image%2020260428082132.png)

* **Preamble and SFD (Start Frame Delimiter)** - used for synchronization and allow devices to be prepared to receive the data in the frame
* **Destination** - Layer 2 address to which the frame is being sent.
* **Source** - Layer 2 address of the device that sent the frame
* **Type** - Layer 3 protocol used in the encapsulated packet, such as IPv4/IPv6. Depending on the version of Ethernet, the Type field can be **Length** field that determines the length of the encapsulated packet.
* **FCS (Frame Check Sequence)** - used by receiving device to detect errors that might be detected during transmission.

### Preamble & SFD

**Preamble**

* Length: 7 bytes (56 bits)
* Alternating 1's and 0's
* Bit pattern: 10101010 * 7
* allows devices to synchronize their receiver clocks

**SFD**

* 'Start Frame Delimiter'
* Length: 1 byte (8 bits)
* Bit pattern: 10101011
* Marks the end of the preamble, and beginning of the rest of the frame

### Destination & Source

They indicate the devices sending and receiving the frame. They consist of destination and source **MAC address** (Media Access Control). MAC Address is a 6 byte (48 bit) address of the physical device. MAC address is assigned to the device at the time of being made.

### Type / Length

They are both 2 bytes (16 bit) field that indicate either the type of length of the encapsulated packet. A value of **1500 or less** in this field indicates the LENGTH of the encapsulated packet (in bytes). A value of **1536 or greater** in this field indicates the TYPE of the encapsulated packet (usually IPv4 or IPv6), and the length is determined via other methods.

### Summary: Ethernet Frame Header bytes

* **Preamble:** 7 bytes
* **SFD:** 1 byte
* **Destination:** 6 bytes
* **Source:** 6 bytes
* **Type:** 2 bytes

### FCS (Frame Check Sequence)

The FCS is the only content in the trailer of an Ethernet frame. It is **4 bytes (32 bits)** in length. FCS detects corrupted data by running a **CRC algorithm** over the received data. CRC means 'Cyclic Redundancy Check'.

### MAC Address

* As mentioned before, a MAC address is different from IP address. You configure IP address through some device configurations using CLI or GUI. A MAC address is a 6 byte (48-bit) physical address assigned to the device when it is made.
* MAC Address is also known as **Burned-In Address** (BIA). This is because MAC address is burned in to the the device when it is made. MAC address is *globally unique* in every device.
* **OUI (Organizationally Unique Identifier)** - the *first 3 bytes* of the MAC address that is assigned to the company making the device.
* The last 3 bytes are unique to the device itself.
* MAC address is written as 12 **hexadecimal** characters

### Simple Network Demonstration

![Pasted image 20260428084913.png](../../_assets/Pasted%20image%2020260428084913.png)

The **F** in F0/1, F0/2, and F0/3 for the switch means "Fast Ethernet", which are 100 Mbps in speed. Each PC have MAC Addresses, and notice that each MAC address is a series of 12 hexadecimal digits separated by periods. Let's break down the MAC Address for PC1

* The **AAAA.AA** is the *OUI* or Organizationally Unique Identifier.
* The last one (**00.0001**) is the unique identifier burned into PC1

Let's say that PC1 wants to send a message to PC2:

![Pasted image 20260428085512.png](../../_assets/Pasted%20image%2020260428085512.png)

PC1 sends a **Unicast** frame. A unicast frame is a frame destined for a *single* target (in this case, the PC2).

![Pasted image 20260428085639.png](../../_assets/Pasted%20image%2020260428085639.png)

Switch 1 (SW1) receives the frame sent by PC1 through its network interface card connected to SW1.

![Pasted image 20260428085924.png](../../_assets/Pasted%20image%2020260428085924.png)

The switch adds the MAC address along with its interface to its internal MAC Address Table. The switch learned about the MAC address itself and is not manually configured to the switch. This is also known as **dynamically learned** MAC address or **Dynamic** MAC address.

The switch learned about the MAC address of PC1 because it received its frame through F0/1 interface. The MAC address of PC1 is added to **Source** information of the Ethernet frame header so this is how the switch dynamically learned it.

But notice that in the MAC Address Table, the switch only know the Source MAC address! It doesn't know where is the MAC Address `AAAA.AA00.0002` even though it is in the **Destination** information of Ethernet frame header. This is also known as **Unknown Unicast frame**. Because the destination MAC and Interface is not in the MAC Address Table, the switch will have to do other method to reach the destination.

![Pasted image 20260428090851.png](../../_assets/Pasted%20image%2020260428090851.png)

The switch will **FLOOD** the frame. FLOOD means to send the frame to all known interfaces except where the switch received its packet on. PC3 will receive the packet, but since the destination MAC is a mismatch, it will simply drops the packet. PC2 receives the packet and process it normally up to OSI stack. However, unless PC2 sends some reply of some sort, it stops there. SW1 will never receive a message from PC2, so it won't be able to populate the MAC Address Table with PC2's MAC Address and interface.

What if PC2 wants to send some data to PC1?

![Pasted image 20260428091744.png](../../_assets/Pasted%20image%2020260428091744.png)

3Notice that the destination and source addresses are now reversed since PC2 is the one sending the frame.

![Pasted image 20260428091839.png](../../_assets/Pasted%20image%2020260428091839.png)

Switch 1 (SW1) receives the frame from PC2 through F0/2 interface. Since SW1 got the information on where does PC2 resides, it will **add** the MAC address and interface of PC2 to the MAC Address Table.

This time, SW1 already knows where to send the data! This is because PC1 MAC address and interface is *already* at the MAC Address Table, so SW1 *doesn't have to flood the frame*. When the destination is already at the MAC address table, this is known as **Known unicast frame**. So, instead of flooding, SW1 will simply **FORWARD** the packet to the destination MAC address.

![Pasted image 20260428092355.png](../../_assets/Pasted%20image%2020260428092355.png)

PC1 now processes the frame up to OSI stack, applies *decapsulation* to read the data.

**NOTE:** In Cisco switches, Dynamic MAC addresses are **removed** from the MAC address table after **5 minutes** of inactivity. If PC1 stops sending some data within 5 minutes, the SW1 will remove its MAC address and interface to clean up the MAC Address Table. If PC1 sends some data after some inactivity, SW1 will simply dynamically learn its MAC address again.
