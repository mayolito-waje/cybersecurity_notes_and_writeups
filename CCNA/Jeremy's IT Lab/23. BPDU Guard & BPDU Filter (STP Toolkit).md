### PortFast & BPDUs

![Pasted image 20260603163814.png](../_assets/Pasted%20image%2020260603163814.png)

* **PortFast** makes a port start in the Forwarding state when it is connected, but it doesn't disable STP on the port.
  * The port will continue to send BPDU every 2 seconds.
* Because end hosts don't run STP and send BPDUs, a PortFast-enabled port shouldn't receive BPDUs.
  * But *what if it does?*

![Pasted image 20260603164011.png](../_assets/Pasted%20image%2020260603164011.png)

Let's say we replaced the end host with another switch and SW1's `G0/1` interface is still in PortFast.

* If a PortFast-enabled port receives an STP BPDU, it will **revert to acting like a regular STP port** (without PortFast).
* In the example above, SW2 have a lower Bridge Priority (4096), so the STP topology changed and SW2 became the new root bridge. SW1's interface changed from designated to root. This can be problematic as it can lead to completely different STP topology.

### BPDU Guard - The Problem

![Pasted image 20260603164407.png](../_assets/Pasted%20image%2020260603164407.png)

The office have a simple LAN of 3 switches. SW1 is the root bridge with all designated ports. SW3 is connected to a Wall jack.

Bob from Accounting is a user in the office. Users do not typically connect directly to switches and usually connect to a wall jack (that is connected to switches). Bob is tired of walking to the printer so he decided to bring his own printer. He also bring his own laptop because the WiFi is slow and he want to directly connect to the network. But the wall jack in the office only have one port. What does he do?

![Pasted image 20260603165014.png](../_assets/Pasted%20image%2020260603165014.png)

Instead of submitting a ticket to the IT department, Bob brought his own switch from home and directly connects it to a wall jack. He then connects the computer, his laptop, and printer directly to the switch. Problem solved.

Bob probably did solved *his* problem. But where is the other problem?

![Pasted image 20260603165202.png](../_assets/Pasted%20image%2020260603165202.png)

Bob's Switch and SW3 will exchange BPDUs! Let's say Bob's switch becomes the root bridge. What will happen?

![Pasted image 20260603165407.png](../_assets/Pasted%20image%2020260603165407.png)

The entire STP topology would change. What if the office deliberately made SW1 the root bridge because it is nearest to the router, so that the path towards the router is efficient?

![Pasted image 20260603165703.png](../_assets/Pasted%20image%2020260603165703.png)

The frames from host connected to SW2, have to take the less efficient path through SW3 instead because the path to SW1 is blocked.

This isn't a disaster. But we don't want unauthorized devices negatively impacting the network!

**The problem:**

* PortFast should only be enabled on ports connected to non-switch devices (end hosts, routers).
  * These devices don't send BPDUs.
* A PortFast-enabled port still sends BPDUs and will operate like a regular STP port if it receives BPDUs from a neighbor.
* If an end user carelessly connects a switch to a port meant for end hosts, it could affect the STP topology.
  * **BPDU Guard** acts as a safeguard against this.

---

### BPDU Guard - The Solution

![Pasted image 20260603170923.png](../_assets/Pasted%20image%2020260603170923.png)

* **BPDU Guard** protects the network from unauthorized switches being connected to ports intended for end hosts.
* It can be configured separately from **PortFast**, but both features are usually used together.
  * They both enhance STP's functionality on ports intended for end hosts.
* A BPDU Guard-enabled port continues to send BPDUs, but if it receives a BPDU it enters the **error-disabled** state. In effect, this disables the port.

### ErrDisable

![Pasted image 20260603172116.png](../_assets/Pasted%20image%2020260603172116.png)

If BPDU Guard-enabled SW3 receives a BPDU on its PortFast-enabled port, the terminal output will look like this:

![Pasted image 20260603172209.png](../_assets/Pasted%20image%2020260603172209.png)

**ErrDisable** is a Cisco switch feature that disables a port under certain conditions, such as BPDU Guard violation. You will learn a few others for the CCNA exam, such as:

* Power Policing violations
* Port Security violations
* DAI (Dynamic ARP Inspection) violations

![Pasted image 20260603172421.png](../_assets/Pasted%20image%2020260603172421.png)

If you connect a switch to a BPDU Guard enabled port, the light indicator will have **solid amber light** (not blinking amber) on Cisco switches. This means it entered **ErrDisable**, so the port can't send or receive data. What if you remove the cable?

![Pasted image 20260603172619.png](../_assets/Pasted%20image%2020260603172619.png)

The **ErrDisable** will still persist on the port even after disconnecting the cable! Usually, the lights will also turn off, but since ErrDisable will persist, the solid amber light will still be present on the port.

So, disconnecting cables from the switch does not solve **ErrDisable** issue. So, how to solve this?

### ErrDisable Recovery

![Pasted image 20260603172937.png](../_assets/Pasted%20image%2020260603172937.png)

To re-enable an err-disabled port, **first solve the underlying issue**.

* If you re-enable the port without fixing the issue, it will just be err-disabled again.
* In the example above, if you re-enable the port again, Bob's switch will just send BPDU again and will err-disabled the port again..

![Pasted image 20260603173123.png](../_assets/Pasted%20image%2020260603173123.png)

Let's disconnect Bob's switch from the wall jack and reconnect the user PC.

You can re-enable an err-disabled port in two ways:

1. Manual: use `shutdown` and `no shutdown` to reset the disabled port.
1. Automatic: `ErrDisable Recovery`

**ErrDisable Recovery** is a feature that automatically re-enables err-disabled ports after a certain period of time.

To view the status of err-disable recovery on the switch, use the command `show errdisable recovery`:

![Pasted image 20260603173718.png](../_assets/Pasted%20image%2020260603173718.png)

Use `errdisable recovery cause <cause>` to enable ErrDisable Recovery for ports disabled by a particular cause.

![Pasted image 20260603173812.png](../_assets/Pasted%20image%2020260603173812.png)

Like mentioned before, always fix the underlying issue first before doing ErrDisable Recovery. If you didn't solve the BPDU Guard issue, the connected switch will send a BPDU again and disable the company switch even after doing ErrDisable Recovery.

### BPDU Filter - The Problem

![Pasted image 20260603174108.png](../_assets/Pasted%20image%2020260603174108.png)

A switch port connected to an end host continues sending BPDUs every 2 seconds regardless of whether PortFast and/or BPDU Guard are enabled.

If the port doesn't connect to a switch, sending BPDUs is unnecessary and undesirable for a couple of reasons:

1. Sending BPDUs uses some bandwidth and processing power on the switch (although it's minimal).
1. BPDUs contain information about the LAN's STP topology. If maximum security is a concern, you should avoid sending this info to user devices.

**BPDU Filter** solves this by preventing a port from sending BPDUs.

---

### BPDU Filter - The Solution

![Pasted image 20260603174922.png](../_assets/Pasted%20image%2020260603174922.png)

**BPDU Filter** stops a port from sending BPDUs. Unlike BPDU Guard, it does not disable the port if it receives a BPDU.

BPDU Filter can be enabled in two ways:

* **Per-port**: `SW3(config-if)# spanning-tree bpdufilter enable`
  
  * The port will not send BPDUs.
  * **The port will ignore any BPDU it receives.** (REMEMBER THIS)
  * In effect, this disables the STP on the port. **USE WITH CAUTION**.
  * \*\*Connecting a switch to a PortFast-enabled port will cause a *temporary Layer 2 loop*. However, connecting a switch to a BPDU Filter-enabled port will cause a *permanent Layer 2 loop!* **
* **Default:** `SW3(config)# spanning-tree portfast [edge] bpdufilter default`
  
  * BPDU Filter will be activated on **all PortFast-enabled ports**. You can use `spanning-tree bpdufilter disable` to disable it on specific ports.
  * The port will not send BPDUs.
  * If the port receives a BPDU, **PortFast and BPDU Filter are disabled, and it operates as a normal STP port**.
  * ***This is the better method. Because it won't ignore any receiving BPDUs which prevent a disaster of having permanent Layer 2 loops***.
