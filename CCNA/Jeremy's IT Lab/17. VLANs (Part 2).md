### Trunk Ports

* In a small network with few VLANs, it is possible to use a separate interface for each VLAN when connecting switches to switches, and switches to routers.
* However, when the number of VLANs increases, this is not viable. It will result in wasted interfaces, and often routers won't have enough interfaces for each VLAN.
* You can use **trunk ports** to carry traffic from multiple VLANs over a single interface.
* Trunk ports are *different* from access ports

Now how does SW1 know which VLAN the traffic belongs to? Both VLAN10 and VLAN30 are allowed to forward and receive traffic between SW1 and SW2 through a single interface.

* Switches will **'tag'** all frames that they send over a trunk link. This allows the receiving switch to know which VLAN the frame belongs to.
* Another name for trunk ports are **tagged ports** while another name for access ports are **untagged ports**.
* Frames sent through an access port aren't tagged, because the interface belongs to the same single VLAN.

### VLAN Tagging

* There are two main trunking protocols: **ISL** (Inter-Switch Link) and **IEEE 802.1Q**.
* ISL is an old Cisco proprietary protocol created before the industry standard IEEE 802.1Q.
* IEEE 802.1Q is an industry standard protocol created by the IEEE (Institue of Electrical and Electronics Engineers).
* You will probably NEVER use ISL in the real world. Even modern Cisco equipment doesn't support it. For the CCNA, you only need to learn 802.1Q.

Here is the regular Ethernet frame header:

![ 500](../_assets/Pasted%20image%2020260521172047.png)

Here is the Ethernet Header with 802.1Q Tag:

![ 500](../_assets/Pasted%20image%2020260521172137.png)

* The 802.1Q tag is inserted between the **Source** and **Type/Length** fields of the thernet frame.
* The tag is 4 bytes (32 bits) in length.
* The tag consists of two main fields: Tag Protocol Identifier (TPID) and Tag Control Information (TCI)
* The TCI consists of three sub-fields.

![ 500](../_assets/Pasted%20image%2020260521172423.png)

### TPID (Tag Protocol Identifier)

* 16 bits (2 bytes) in length
* Always set to a value of 0x8100 (hexadecimal). This indicates that the frame is 802.1Q-tagged.

### PCP (Priority Code Point)

* 3 bits in length
* Used for Class of Service (CoS), which prioritizes important traffic in congested networks.

### DEI (Drop Eligible Indicator)

* 1 bit in length
* Used to indicate frames that can be dropped if the network is congested.

### VID (VLAN ID)

* 12 bits in length
* Identifies the VLAN the frame belongs to.
* 12 bits in length = 4096 total VLANs (2<sup>12</sup>), range of 0 - 4095.
* VLANs 0 and 4095 are reserved and can't be used, therefore, the actual range of VLANs is 1 - 4094.
* Cisco's proprietary ISL also has a VLAN range of 1 - 4094.

### VLAN Ranges

* The range of VLANs (1 - 4094) is divided into two sections:
  * Normal VLANs: 1 - 1005
  * Extended VLANs: 1006 - 4094
* Some older devices cannot use the extended VLAN range, however it's safe to expect that modern switches will support the extended VLAN range.

### Native VLAN

* 802.1Q has a feature called the **native VLAN**. ISL does not have this feature.
* The native VLAN is VLAN 1 by default on all trunk ports, however this can be manually configured on each trunk port.
* The switch does not add an 802.1Q tag to frames in the native VLAN. When a switch receives an untagged frame on a trunk port, it assumes the frame belongs to the native VLAN.
* **It's very important that the native VLAN matches!**

Let's assume that we set the native VLAN to VLAN10 on both switches:

![ 500](../_assets/Pasted%20image%2020260521174014.png)

The traffic will flow smoothly and no issues. But how about if there is a native VLAN mismatch? Let's assume we set the native VLAN to 10 on SW2, but set the native VLAN to 30 on SW1:

![ 500](../_assets/Pasted%20image%2020260521174208.png)

The switch will not forward the frame to VLAN30, but drop it!

### Trunk Configuration

![ 500](../_assets/Pasted%20image%2020260521174331.png)

Let's use the above setup to set our trunk ports configuration.

![Pasted image 20260521174519.png](../_assets/Pasted%20image%2020260521174519.png)

`switchport mode trunk` - manually configure the interface as a trunk.

However, there is an error. The command was rejected. This is quite tricky. Many modern switches do not support Cisco's ISL at all. They only support 802.1Q (dot1q). However, switches that do support both (like the one in the example above) have a trunk encapsulation of 'Auto' by default. To manually configure the interface as a trunk port, you must first set the encapsulation to 802.1Q or ISL. On switches that only support 802.1Q, this is not necessary. After you set the encapsulation type, you can then configure the interface as a trunk.

`switchport trunk encapsulation ?` - set the trunk encapsulation of the interface. `negotiate` will set the trunk encapsulation to `Auto` so use `dot1q`.

![ 500](../_assets/Pasted%20image%2020260521175130.png)

`show interfaces trunk` is a command to confirm your trunk ports. `Mode` set to `on` means that the interface was manually configured as a trunk. We can also see that the `Encapsulation` is set to `802.1q` and the `Status` is `trunking`. The `Native vlan` is also set to `1`.

We can see that VLAN 1 to 4094 is allowed on trunk. But it is best practice to limit what VLANs are allowed on trunk port:

![ 500](../_assets/Pasted%20image%2020260521175620.png)

`WORD` allows us to simply list the VLANs allowed.

![ 500](../_assets/Pasted%20image%2020260521175734.png)

After using `switchport trunk allowed vlan 10,30`, notice that the VLANs allowed on trunk is now only set to VLAN 10 and VLAN 30.

### Router on a Stick (ROAS)

![ 500](../_assets/Pasted%20image%2020260522082859.png)

Now you are wondering about the router. In the previous lesson, we configured each connected interface to an IP address which is the default gateway for each subnet the interfaces belongs to. However, now that we are using only **one interface**, we have to use *sub-interfaces* on R1.

This is also known as **Router on a Stick** (ROAS) because we only use one physical cable on a router to handle inter-VLAN routing and it looks like a stick. In this case, the interface connected to R1 is `G0/0` and is connected to `G0/1` interface on SW2. We can divide the single physical interface into 3 sub-interfaces which will allow us to do inter-VLAN routing.

Using ROAS, the network will look like this:

![ 500](../_assets/Pasted%20image%2020260522083529.png)

We use `G0/0.10` for `VLAN10`, `G0/0.20` for `VLAN20`, and `G0/0.30` for `VLAN30`. These 3 logical sub-interfaces are actually from one physical interface, but they can operate like 3 interfaces. Note that we don't need to do any configurations on SW2, we only configure on router since we already configured `G0/1` on SW2 as a trunk port!

![ 500](../_assets/Pasted%20image%2020260522084031.png)

* First, we go to interface `g0/0` and apply `no shutdown` to enable it.
* Then we go to config-sub mode by entering command `interface g0/0.10` to go to sub-interface to configure VLAN10.
* The subinterface number **does not** have to match the VLAN number. However, it is **highly recommended** that they do match, to make it easier to understand.
* `encapsulation dot1q 10` tells the router to treat any arriving frames tagged with a specific VLAN number (in this case, VLAN10) as if they arrived on this sub-interface (the interface config-sub is operating at).
* With `encapsulation dot1q 10`, if any arriving frames are tagged with VLAN10, the router will treat as it arrived from interface `g0/0.10`. The router will also tag any frames leaving with this sub-interface with VLAN10 using `dot1q`.
* Then we do the same command of applying IP address and default gateway on the sub-interface.

![ 500](../_assets/Pasted%20image%2020260522085110.png)

We then do the same with other two sub-interfaces: go to that sub-interface, encapsulate it with dot1q, then set IP address and netmask. Remember that that it is recommended that sub-interface number should match the VLAN number we want to configure the sub-interface at.

![ 500](../_assets/Pasted%20image%2020260522085313.png)

As you can see, `show ip interface brief` confirms that each of the sub-interface appears as well as the physical interface, although the actual physical interface itself have no IP address!

![ 500](../_assets/Pasted%20image%2020260522085450.png)

In `show ip route`, notice that the local and connected routes are configured the same, treating each sub-interfaces as normal interfaces connected on a router! If a router receives a packet with destination IP of `192.168.1.64/26`, it will tag the leaving packet with VLAN20!

### ROAS Review

* ROAS is used to route between multiple VLANs using a single interface on the router and switch.
* The switch interface is configured as a regular trunk.
* The router interface is configured using **subinterfaces**. You configure the VLAN tag and IP address on each subinterface.
* The router will behave as if frames arriving with a certain VLAN tag have arrived onthe subinterface configured with that VLAN tag.
* The router will tag frames sent out of each subinterface with the VLAN number where the destination IP resides.

Let's say that PC from VLAN10 tries to reach PC from VLAN30.

![ 500](../_assets/Pasted%20image%2020260522090030.png)

![ 500](../_assets/Pasted%20image%2020260522090105.png)

R1 will receive the frame through `G0/0` interface, but it will treat the frame as it came from `G0/0.10` interface because of the `VLAN10` tag. The destination is in the subnet `192.168.1.128/26` which is connected to R1's `G0/0.30` sub-interface.

![ 500](../_assets/Pasted%20image%2020260522090322.png)

R1 will then send the frame to the destination subnet. It will tag the frame with `VLAN30` because this is the VLAN where the subnet resides. The PC from VLAN30 will receive it after some hops.
