### What is a LAN?

* By simple definition, a LAN is a group of devices (PCs, servers, routers, switches, etc.) in a single location (home, office, etc.)
* A more specific definition: A LAN is a single **broadcast domain**, including all devices in that broadcast domain.

### LANs/Broadcast Domains

![ 500](../_assets/Pasted%20image%2020260520083525.png)

A **broadcast domain** is a group of devices that will receive the broadcast frame (FF:FF:FF:FF:FF:FF) if one the devices in that domain sends it. Even though R1 will only have R2 as the device that will receive the broadcast frame, it is still considered a broadcast domain despite the absence of switches.

### Virtual Local Area Networks (VLANs)

Let's say that a company has a small LAN that uses a network IP of `192.168.1.0/24`

![ 500](../_assets/Pasted%20image%2020260520084013.png)

Note that this is not the best setup. For both security and performance purposes, it is best to separate them into subnets.

Let's say one device from Engineering department sent a broadcast frame. What will happen? Well all of the devices will receive the broadcast frame since they are under the same LAN.

![ 500](../_assets/Pasted%20image%2020260520084306.png)

This is problematic, however. There would be unnecessary amount of flooding.

* For **performance** reasons, this will have lots of unnecessary broadcast traffic that can reduce network performance.
* For **security** reasons, even with the same office, you want to limit who has access to what. You can apply security policies on a router/firewall. Because this is one LAN, PCs can reach each other directly, without traffic passing through the router. So, even if you configure security policies, they won't have any effect.

Let's separate them into different subnets. We also have to configure different interfaces for each subnet in a router (for now, there are more efficient methods we will learn in the future):

![ 500](../_assets/Pasted%20image%2020260520084836.png)

Let's say that PC1 will send a data to PC2. PC1 knows that PC2 is in another subnet so PC1 will send the frame to its default gateway (R1):

![ 500](../_assets/Pasted%20image%2020260520085049.png)

R1 will then forward the frame back to the switch then the switch will forward the frame to PC2 (we are assuming that ARP process has been done so we won't go into details into ARP).

![ 500](../_assets/Pasted%20image%2020260520085230.png)

So, instead of PC1 being able to directly send traffic to PC2, we set up some security and config policies that will force what traffics are allowed to pass through the subnets.

Now, what if PC1 sends an unknown unicast or broadcast frame? When switch receives a broadcast frame, it will flood the frame to all interfaces except where it was received on. Let's say that PC1 sends a broadcast frame intended for the engineering department (note that destination IP is `192.168.1.63` which is the broadcast address of Engineering department's subnet):

![ 500](../_assets/Pasted%20image%2020260520090114.png)

Oops! The switch floods the frame to all of the available devices even on devices outside the subnet! This is because the switch doesn't know about IP addresses and subnets! It *only works at layer 2* and it will flood every broadcast frames to all available interfaces no matter what subnet does that interface belong to.

Although we separated the three departments into three subnets (Layer 3), they are **still in the same broadcast domain** (Layer 2). Each department can buy a separate switch to solve this problem, which is not an elegant solution, and is also expensive.

### Sementing at Layer 2 (VLANs)

We can segment the broadcast domain at layer 2 using VLANs. For each department in the company, we can set up a separate VLAN for them:

![ 500](../_assets/Pasted%20image%2020260520090924.png)

How do we configure VLAN? We configure them on the switches. We configure the switch interfaces to be in a specific VLAN, and the end hosts connected to that interfaces is part of that configured VLAN. Switch will treat each VLAN as separate LANs and will not forward traffic on VLAN#2 if the frame comes from end host at VLAN#1. A switch WILL NOT forward traffic between VLANs, including **broadcast/unknown unicast** traffic.

So, after setting up VLAN, if PC1 sends a broadcast frame, the switch will not forward the traffic on a different VLAN!

![ 500](../_assets/Pasted%20image%2020260520091503.png)

If PC1 however wants to send data to PC2, it will still works by forwarding the data through router.

![ 500](../_assets/Pasted%20image%2020260520091816.png)
![ 500](../_assets/Pasted%20image%2020260520091857.png)

Notice that it must pass through R1 first for the data to reach PC2, the switch does not perform **inter-VLAN routing**. It must send the traffic through the router.

![ 500](../_assets/Pasted%20image%2020260520092355.png)

The switch will not forward the traffic to an interface in a different VLAN. It must pass through R1 first, and R1 will use an interface connected to the same VLAN to forward the data so that when the switch receives it, it will flood/forward it in the same VLAN. Even if PC1 and PC2 are under the same subnet, the switch won't directly forward the traffic as long as they are still under a different VLANs.

### VLANs Summary

VLANs...

* are configured on switches on a **per-interface** basis.
* **logically** separate end hosts at Layer 2.

Switches do not forward traffic directly between hosts in different VLANs.

### VLAN Configuration

![ 500](../_assets/Pasted%20image%2020260520093904.png)

**`show vlan brief`** - displays VLANs that exist on the switch and which interfaces are connected to that VLAN.

![ 500](../_assets/Pasted%20image%2020260520094851.png)

If you don't configure VLAN yet, all interfaces are configured in **VLAN 1** by default. Under the default VLAN are for VLAN 1002 to VLAN 1005, used for fddi and token-ring. These are old technologies and won't be covered in CCNA.

VLANs 1,  1002-1005 exist by default and **cannot be deleted**.

To setup VLAN on interfaces, we can use a combination of commands:

![ 500](../_assets/Pasted%20image%2020260520095331.png)

Here, we configured the interface range from `Gi1/0` to `Gi1/3`.

We used `switchport mode access` command to set the interface as an access port. An access port is a switchport which belongs to a single VLAN, and usually connects to end hosts like PCs. Switchports which carry multiple VLANs are called 'trunk ports'. We will learn them in the next lesson!

A switchport connected to an end host, should have access port enabled by default, but it is a good practice to set them up explicitly.

`switchport access vlan 10` is the command to actually assign the switchport to a VLAN. If the VLAN does not exist, it was created automatically.

We also did the same config on interface `Gi2/0 - 2` and `Gi3/0 - 3`.

Now let's use `show vlan brief` again to check for freshly made VLANs:

![ 500](../_assets/Pasted%20image%2020260520100138.png)

We now can see the configured VLANs and which interfaces are associated with that VLAN. Notice the names of the VLANs. We can change them to make them understandable:

![ 500](../_assets/Pasted%20image%2020260520100313.png)

By the way, `vlan <number>` is also the command to *manually create a VLAN*. But in our case, it was already automatically created when we assigned the interaces. Use `name <name>` on the config-vlan mode to change the name of the VLAN.

Now let's look at our changes:

![ 500](../_assets/Pasted%20image%2020260520100543.png)

Now the VLAN names are more understandable and which departments use that VLAN!

Now after configuring the VLANs, if PC2 pings `255.255.255.255` which will use destination MAC of `FF:FF:FF:FF:FF:FF`, only the end hosts under PC2's VLAN will receive it.
