### PortFast - the problem

![Pasted image 20260602090333.png](../_assets/Pasted%20image%2020260602090333.png)

* When an end host such as PC connects to a switch port, it will be frustrating because the end host cannot send/receive any data yet even though the switch port is in **up/up** state.
* This is because while the switch port is a **designated port**, it would need to enter **Listening** state then **Learning** state first before it can enter **Forwarding** state
* Listening and Learning state both takes 15 seconds so it will take the switch port at least **30 seconds** before the switch port enters Forwarding state.
* Listening and Learning state are both important to prevent **Layer 2 loops** *between switches* but it is unnecessary between switch and end hosts.
* Keeping the Listening and Learning state on the connection between switch and end host can lead to poor user experience.

---

### STP Demo - Cisco Catalyst

![ 500](../_assets/Pasted%20image%2020260602090934.png)

If you connect an end host to an Ethernet port on the real switch (such as the Cisco Catalyst above), notice the amber lights after you connect the black cable with an RJ45 connector. This is a link light to indicate status. The two blinking green status on the bottom right indicates that it is active and is both sending and receiving data.

The amber lights indicator from Port 1 means that the switch port is up and can receive data from the PC, but is being blocked by STP, which means the PC cannot access the Internet.

![ 500](../_assets/Pasted%20image%2020260602091343.png)

Finally, after 30 seconds, the blinking light turns green which means it is now in Forwarding state and can both send and receive data.

---

### PortFast - the solution

![ 500](../_assets/Pasted%20image%2020260602091530.png)

When **PortFast** is configured on the port, the port immediately enters the **Forwarding** state when connected to another device. PortFasts bypasses **Listening/Learning** and can send and receive data right away.

This solves the poor user experience regarding not being able to access the network right away after connecting the end host to a switch.

Like with the demo above, if you connect and end host to a PortFast enabled port, the light will immediately blink green which indicates that the switch can send and receive data from end host right away.

---

### PortFast Configuration

We already know that we should not enable PortFast on connection with other switches. But with end hosts, we can do it. To configure PortFast:

* In interface config mode:
  `SW1 (config-if)# spanning-tree portfast`
  This enables PortFast only on the individual interface

* In global config mode:
  `SW1 (config)# spanning-tree portfast default`
  This enables PortFast on **all access ports**.

Remember, in modern networks, connections between switches are almost always **trunk** links. Connections to end hosts are almost always **access** links.

---

### PortFast Configuration: Per-Port

![Pasted image 20260602092512.png](../_assets/Pasted%20image%2020260602092512.png)

Commands used:

* `interface g0/1`: Enter GigabitEthernet 0/1 interface
* `spanning-tree portfast`: Enable PortFast specifically on that interface (which connects to end host)
* `show spanning-tree interface g0/1 detail`: Check the detailed STP configuration of `g0/1`. We can see that the PortFast we configured is in **PortFast Edge** mode. We do not need to know **PortFast Network** as it is outside of CCNA topic.

---

### PortFast Configuration: Default

![Pasted image 20260602095511.png](../_assets/Pasted%20image%2020260602095511.png)

* The warning regarding enabling PortFast on all interfaces is a bit misleading. Using `spanning-tree portfast default` only applies on **access ports**.
* However, for the peace of mind, you can use `spanning-tree portfast disable` on a specific interface to disable PortFast there.
* On a trunk connection, even after using `spanning-tree portfast default`, the switch port won't enter PortFast Edge because PortFast won't have an effect on a trunk link as you can see in the output of `show spanning-tree interface g0/1 detail` (where SW1's `g0/1` interface is a trunk port).

---

### PortFast on Trunk Ports

![Pasted image 20260602100103.png](../_assets/Pasted%20image%2020260602100103.png)

The standard PortFast configuration commands only enable PortFast on access ports.

* Per-port: `SW1 (config-if)# spanning-tree portfast`
* Default: `SW1 (config)# spanning-tree portfast default`

In some cases, you might want to enable PortFast on a trunk port:

* A port connected to a virtualization server with virtual machines (VMs) in different VLANs. Virtualization servers often uses trunk ports.
* A port connected to a router via router-on-a-stick (ROAS). It is safe to enable PortFast when connecting to routers as it won't trigger L2 loops.

To configure PortFast on a trunk link, it can only be configured per-port in interface config mode:

* `SW1 (config-if)# spanning-tree portfast trunk`

![Pasted image 20260602100429.png](../_assets/Pasted%20image%2020260602100429.png)

We can now see in interface STP detail that the port is in **portfast edge trunk mode**.
