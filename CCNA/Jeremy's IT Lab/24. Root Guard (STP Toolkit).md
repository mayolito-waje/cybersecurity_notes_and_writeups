**Root Guard** prevents a port from becoming a Root Port by disabling it if superior BPDUs are received, thereby enforcing the current Root Bridge.

### Root Bridge Placement

![Pasted image 20260603203907.png](../_assets/Pasted%20image%2020260603203907.png)

STP prevents loops by electing a root bridge and ensuring that each other switch has only **one valid path** to reach it.

You shouldn't randomly select the root bridge. Some things you should consider include:

* Optimal traffic flow
  * minimize latency
  * minimize congestion
* Stability and reliability

![Pasted image 20260603204045.png](../_assets/Pasted%20image%2020260603204045.png)

In this case, SW1 is picked as a root bridge. PCs from different LANs usually don't communicate with each other and they often communicate more outside the network (cloud/Internet). SW1 is a good candidate because it sits between the router and the LANs, allowing for end hosts in different LANs to access the Internet more efficiently.

![Pasted image 20260603205033.png](../_assets/Pasted%20image%2020260603205033.png)

If you choose SW3 as the root bridge, the path from SW3 to the router is still the same. But the traffic from SW2 will have less efficient path because the path directly to SW1 is blocked. This can cause some latency issues (although very minimal). However there will be some instance where the traffic between SW1 and SW3 will be congested due to some busy operations so it is a thing to consider.

Another consideration when selecting the root bridge is stability and reliability. Once you select the root bridge that provides an efficient path for traffic in the LAN, you want to ensure that the switch is up and operational for as long as possible. It is always recommended to use a new switch instead of old ones for better reliability when setting them up as root bridge especially if there is some budget constraints and you cannot buy more switches.

### Root Guard - The Problem

![Pasted image 20260603205534.png](../_assets/Pasted%20image%2020260603205534.png)

Within your own LAN, you can easily control the root bridge by setting its priority to **0**. But there are cases where you might connect your LAN to other switches outside of your direct control. For example, in the example above, a service provider offering **Metro Ethernet** service to customers. Metro Ethernet is often used to connect sites withing a MAN (Metropolitan Area Network) such as connecting to other offices within the city.

Even if you set your root bridge's priority to 0, its role can be taken by another switch with a lower MAC address. In the example above, the root bridge from Customer LAN will take over and also become the root bridge of the Service Provider LAN because of lower MAC address (despite both root bridge having priority of 0).

![Pasted image 20260603210017.png](../_assets/Pasted%20image%2020260603210017.png)

With no safeguard in place, SW1, SW2, and SW3 accept SW6 as the root bridge, affecting the service provider's STP topology.

* Frames from SW3 to SW1 must take a detour through the customer's LAN just to talk with each other!

**Root Guard** can be configured to protect your STP topology by preventing your switches from accepting **superior BPDUs** from switches outside of your control.

* **Superior BPDU** = a BPDU that is superior in the STP algorithm (e.g. claiming a better root bridge ID).

---

### Root Guard - The Solution

![Pasted image 20260603210458.png](../_assets/Pasted%20image%2020260603210458.png)

If you want to ensure that the root bridge remains in your LAN, you can configure **Root Guard** on the ports connected to switches outside of your control.

* SW2 G0/2, SW3 G0/2

To enable Root Guard on the port, use the command:

* `SW2(config-if)# spanning-tree guard root`
* **Note:** There is no command to enable it by default from global config mode, so you have to set them per port.

![Pasted image 20260603211015.png](../_assets/Pasted%20image%2020260603211015.png)

If a Root Guard-enabled port receives a BPDU, it will enter the **Broken** (*Root Inconsistent*) state, effectively disabling it.

* The port will not be able to forward data frames and will discard any frames it receives.
* SW1, SW2, and SW3 won't accept SW6 as the root bridge.
* So even though Service Provider and Customer LAN are physically connected, they can't communicate.

![Pasted image 20260603211310.png](../_assets/Pasted%20image%2020260603211310.png)

Normally, in STP, there should be only one designated port per collision domain. But since SW2's `G0/2` and SW3's `G0/3` interface are disagreeing from the root guard and blocking the link, it disagrees in being a root port so it is a special case of having both connections being a designated port.

![Pasted image 20260603211927.png](../_assets/Pasted%20image%2020260603211927.png)

To re-enable a port disabled by Root Guard, you must solve the issue that disabled the port.

* The disabled port must stop receiving superior BPDUs.
* To stop receiving superior BPDUs, the service provider should tell the customer to increase the priority value of their switch.

Once the customer increased the priority value of the switch (priority of 4097, which is highlighted in red):

![Pasted image 20260603212651.png](../_assets/Pasted%20image%2020260603212651.png)

Once the superior BPDUs received by SW2 `G0/2` and SW3 `G0/2` age out, the ports will automatically be re-enabled.

* A BPDU's Max Age is 20 seconds by default.

SW4, SW5, and SW6 will finally accept SW1 as their root bridge and the STP topology will also be updated.

### Where to configure root guards?

![Pasted image 20260603213709.png](../_assets/Pasted%20image%2020260603213709.png)

* You configure root guards on interfaces that connects to the customer's network.
* In this case SW2's `G0/2` and SW3's `G0/2` have physical connection to the customer network, so this is where you set up the root guard.
* You shouldn't always configure root guards on interfaces connecting to other networks.
* In the above example, the customer SHOULD not configure root guards on their switch's interface that connect physically to service provider's network.
* If you configure root guard on SW4 and SW5, those switches won't accept SW1's BPDUs and block the connection link, which means service provider and customer won't be able to communicate with each other, which erases the point on why the customer need/want to connect with the service provider.
