### EtherChannel

![ 500](../_assets/Pasted%20image%2020260606204047.png)

* EtherChannel groups multiple interfaces together to act as a single interface.
* STP will treat this group as a single interface.

![ 500](../_assets/Pasted%20image%2020260606204509.png)

* If one of the hosts connected to ASW1 sends a broadcast frame, ASW1, won't send 4 copies of the frame to DSW1! And DSW1 won't forward the interfaces back to its other interfaces connected to ASW1.
* This is because the link between ASW1 and DSW1 are treated as a single interface!
* **Traffic using the EtherChannel will be load balanced among the physical interfaces in the group. An algorithm is used to determine which traffic will use which physical interface.**

![ 500](../_assets/Pasted%20image%2020260606204617.png)

* Think of it like this: the 4 connected interfaces are 1 Gbps each.

* With EtherChannel, you treat the 4 interfaces as single interfaces, but you add all their bandwidths!

* This effectively gives you a virtual 4 Gbps interface, which makes it faster! So EtherChannel will help in not wasting bandwidths of connected interfaces.

* Some other names for an EtherChannel are:
  **PortChannel**
  **LAG (Link Aggregation Group)**

### EtherChannel Load-Balancing

![ 500](../_assets/Pasted%20image%2020260606205315.png)
![ 500](../_assets/Pasted%20image%2020260606205539.png)

* EtherChannel load balances based on **flows**.
* A flow is a communication between two nodes in the network.
* Frames in the same flow will be forwarded using the same physical interface.
* If frames in the same flow were forwarded using different physical interfaces, some frames may arrive at the destination out of order, which can cause problems.
* You can change the inputs used in the interface selection calculation:
  - **Source MAC**
  - **Destination MAC**
  - **Source AND Destination MAC**
  - **Source IP**
  - **Destination IP**
  - **Source AND Destination IP**

![ 500](../_assets/Pasted%20image%2020260606210014.png)

* `show etherchannel load-balance` - show EtherChannel load balancing configuration
* `srd-dst-ip` means it uses **source and destination IP** address as input. For example, all the traffic coming from `10.0.0.1` with destination IP of `10.0.0.2` will always use the same interface within the EtherChannel.
* The last information shows the detailed load balancing addresses used per protocol:
  * IPv4 and IPv6 will be load balanced using source and destination IP address.
  * If an IP packet isn't encapsulated in the Ethernet frame, there isn't IP address info to use to be used in load balancing, so the source and destination MAC will be used instead.

![ 500](../_assets/Pasted%20image%2020260606210735.png)

* `port-channel load-balance ?` - change the load balancing method. Enter global config mode to setup.

![ 500](../_assets/Pasted%20image%2020260606210932.png)

* The sample switch have a lot of options when configuring EtherChannel load balancing method.
* You can use source and destination IP address (or both).
* You can also use source and destination MAC address (or both).

### EtherChannel Protocols: PAgP, LACP, Static

There are three methods of EtherChannel configuration on Cisco switches:

* PAgP (Port Aggregation Protocol)
  * Cisco proprietary
  * Like DTP on trunks, dynamically negotiates creation/maintenance of EtherChannel
* LACP (Link Aggregation Control Protocol)
  * Industry standard (IEEE 802.3ad)
  * Like DTP on trunks, dynamically negotiates creation/maintenance of EtherChannel
* Static EtherChannel
  * Interfaces are statically configured to form an EtherChannel
  * No protocol to determine when to form EtherChannel
  * Not really ideal as best practice

LACP is the preferred method of configuring EtherChannel. Up to 8 interfaces can be formed into a single EtherChannel

* LACP allows up to 16, but only 8 will be active, the other 8 will be in standby mode.

---

### EtherChannel Configuration: PAgP

![ 500](../_assets/Pasted%20image%2020260607085121.png)
![ 500](../_assets/Pasted%20image%2020260607085406.png)

* It is better to use `interface range` when configuring EtherChannel
* `channel-group <channel-number> mode <mode>` - to configure an EtherChannel on the local switch. Just like with trunk port negotiation, PAgP EtherChannel will either try to actively negotiate EtherChannel or not:
  * **auto + auto** - will not form an EtherChannel
  * **desirable + auto** - will form an EtherChannel
  * **desirable + desirable** - will form an EtherChannel
* In this case, we set the `channel-group` to **1** and the mode to `desirable`.

![ 500](../_assets/Pasted%20image%2020260607085826.png)

* After creating a `channel group`, you can see the created virtual interface using `show ip interface brief` and it has an interface name of `Port-channel1`.
* The `channel-group` number has to match for member interfaces on the **same** switch.
* However, it **doesn't** have to match the channel-group number on the other switch.
* For example, `channel-group 1` on ASW1 can form an EtherChannel with `channel-group 2` on DSW1.

---

### EtherChannel Configuration: LACP

![ 500](../_assets/Pasted%20image%2020260607090343.png)
![ 500](../_assets/Pasted%20image%2020260607090335.png)

* The configuration is almost the same on PAgP. The mode names are just different:
  * **active + active** - will form an EtherChannel
  * **active + passive** - will form an EtherChannel
  * **passive + passive** - will not form an EtherChannel
* Remember that even if the EtherChannel is not actively formed (passive + passive), the `port-channel` interface will still be created. It just won't function as an EtherChannel, unless one side is in active mode.

---

### EtherChannel Configuration: Static

![ 500](../_assets/Pasted%20image%2020260607090707.png)
![ 500](../_assets/Pasted%20image%2020260607090743.png)

* To configure EtherChannel statically, use the mode `on`.
* **Note:** `on` mode only works with `on` mode.
* **on + desirable** or **on + active** will not successfully form an EtherChannel.

### Manually Configure Negotiation Protocol

![ 500](../_assets/Pasted%20image%2020260607091225.png)
![Pasted image 20260607091052.png](../_assets/Pasted%20image%2020260607091052.png)

* `channel-protocol <mode>` manually configures the EtherChannel negotiation protocol the interface members should use.
* It's not used that much because you can automatically configure the Ethernet negotiation with the command `channel-group <channel-number> mode <mode>`.
* If you set up the `channel-protocol` to `lacp`, using `channel-group 1 mode desirable` or `channel-group 1 mode on` won't work because of the protocol mismatch.
* Since we manually configured the EtherChannel to use LACP, configuring the channel group with command `channel-group 1 mode active` will work.

![ 500](../_assets/Pasted%20image%2020260607091800.png)

* After configuring the port channel and the EthernetChannel negotiation, we can now treat it as a regular interface!
* The above command is used to setup the virtual interface after configuring LACP, but the command is the same whether you configure EtherChannel with PAgP or statically.
* The command above set the virtual interface as a trunk port using 802.1Q encapsulation.

![Pasted image 20260607092209.png](../_assets/Pasted%20image%2020260607092209.png)![Pasted image 20260607092325.png](../_assets/Pasted%20image%2020260607092325.png)

* In running configuration, you can see that the trunk encapsulation config on an EtherChannel is **both** reflected on created port-channel and the individual interfaces that are member on that port channel!
* They are automatically configured even if we didn't manually configured them.

### EtherChannel Requirements

![ 500](../_assets/Pasted%20image%2020260607092828.png)

* Member interfaces **must have matching configurations** for them to be included in EtherChannel:
  * Same duplex (full/half)
  * Same speed
  * Same switchport mode (access/trunk)
  * Same allowed VLANs/native VLAN (for trunk interfaces)
* If an interface's configurations do not match the others, it will be excluded from the EtherChannel.

### EtherChannel Verification

![ 500](../_assets/Pasted%20image%2020260607131119.png)

* Use `show etherchannel summary` to verify EtherChannel status.
* There are different flags to show status of the EtherChannel interfaces.
* Under `Port-channel` column, port channel **1** has `SU` flags:
  * `S`: means it is a Layer 2 EtherChannel. **S** stands for switchport.
  * `U`: means the EtherChannel is active and in use.
* Under `Ports` column, interfaces in group channel 1 have flag `P`:
  * `P`: the interfaces are bundled in port-channel.

![ 500](../_assets/Pasted%20image%2020260607131630.png)

* If we `shutdown` port channel 1, and check the EtherChannel status, the outcome will be different:
  * `D`: means it is down and inactive

![ 500](../_assets/Pasted%20image%2020260607131816.png)

* Let's change `G0/0` to an access port. Remember that the configured EtherChannel is supposed to be a trunk port.
* `G0/0` now has a different flag if you check the EtherChannel status:
  * `s` (lowercase): means the interface is suspended and does not participate in EtherChannel.

![ 500](../_assets/Pasted%20image%2020260607132131.png)

* Another useful command to verify EtherChannel status is `show etherchannel port-channel`.
* You can see the number of ports participating in EtherChannel and also the protocol (in this case, uses LACP).
* One difference that is only shown in `show etherchannel port-channel` is `EC state`.
  * The EtherChannel in group channel 1 is configured with the command `channel-group 1 mode active`.
  * The `Active` status in `EC state` is because we set the mode to `active` in group channel.

![ 500](../_assets/Pasted%20image%2020260607132531.png)

* In `show spanning-tree` command, the individual interfaces are not listed.
* This is because in STP, it treats the EtherChannel as a single interface which is shown as interface `Po1`.
* Instead of blocking 3 of the ports, STP will treat them as one interface and also effectively avoid Layer 2 loops.

### Layer 3 EtherChannel

![ 500](../_assets/Pasted%20image%2020260607140243.png)

* EtherChannel can also be configured at Layer 3
* Let's change the topology so and replace switches with multilayer switches.
* Let's also use a Layer 3 instead of Layer 2 connection and assign IP address to each virtual interface.

![ 500](../_assets/Pasted%20image%2020260607140623.png)

* To enable Layer 3 routing on interfaces, let's use `no switchport` command on group of interfaces (let's use channel group 1 for this EtherChannel group).
* To configure LACP negotiation on channel group 1, use the command `channel-group 1 mode active`.

![Pasted image 20260607140846.png](../_assets/Pasted%20image%2020260607140846.png)

* The configurations on running config file is automatically updated.
* All interfaces belonging to `Port-channel1` will have `no switchport` enabled automatically.

![ 500](../_assets/Pasted%20image%2020260607141023.png)

* To assign an IP address, let's assign it at the port channel.
* An IP addres of `10.0.0.1/30` is configured on `Port-channel1` of ASW1.

![ 500](../_assets/Pasted%20image%2020260607141428.png)

* In `show etherchannel summary`, the flag of `po1` will be `R` instead of `S`. `R` means Layer 3 so this means this port channel uses routed port.

![ 500](../_assets/Pasted%20image%2020260607141635.png)

* In `show ip interface brief`, `Port-channel1` has an assigned IP, which means it works at Layer 3.
* ASW1 and DSW1 is connected through `Port-channel1`.
* Since `Port-channel1` uses routed port, the connection between ASW1 and DSW1 **doesn't use STP**.
