### Spanning Tree Versions

![Pasted image 20260605100509.png](../_assets/Pasted%20image%2020260605100509.png)

Cisco hasn't developed their own version of MSTP yet and use the standard 802.1s. Use MSTP for large networks. But form small to medium networks, Rapid PVST+ is a good method to use for Cisco switches. We will focus on Rapid PVST+ in this lesson.

---

### RSTP Intro

**Cisco's Summary:**

 > 
 > RSTP is not a timer-based spanning tree algorithm like 802.1D. Therefore, RSTP offers an improvement over the 30 seconds or more that 802.1D takes to move a link to forwarding. The heart of the protocol is a new bridge-bridge handshake mechanism, which allows ports to move directly to forwarding.

Back then, the 30-50 seconds delay before switches can enter forwarding state is acceptable, but now it isn't. The difference between classic 802.1D and the more modern 802.1w is with classic STP, the timer is the indicator when will the switch move to forwarding state. With RSTP, the switches can negotiate with each other and depending on the conditions met, the switch can immediately move its ports to forwarding state.

---

### STP/RSTP Similarities

* RSTP serves the same purpose as STP, blocking specific ports to prevent Layer 2 loops.
* RSTP elects a root bridge with the same rules as STP.
* RSTP elects root ports with the same rules as STP.
* RSTP elects designated ports with the same rules as STP.

### RSTP Port States

Rapid STP further simplified port states. The **Blocking** and **Disabled** states are combined into one state, and the **Listening** state was discarded.

![ 500](../_assets/Pasted%20image%2020260605101937.png)

The combined Blocking and Disabled state have become **Discarding** state. Here is the new table:

![ 500](../_assets/Pasted%20image%2020260605102132.png)

* If a port is administratively disabled (`shutdown` command), the port will be in **discarding** state.
* If a port is enabled but blocking traffic to prevent Layer 2 loops, the port will be in **discarding** state.

---

### RSTP Port Roles

* The **root port** role remains unchanged in RSTP.
  * The port that is closes to the root bridge becomes the root port for the switch.
  * The root bridge is the only switch that doesn't have a root port.
* The **designated port** role remains unchanged in RSTP.
  * The port on a **segment** (collision domain) that sends the best BPDU is that segment's designated port (only one per segment).
* The **non-designated port** role is split into two separate roles in RSTP:
  * The **alternate port** role
  * The **backup port** role

---

### RSTP: Alternate Port Role (+UplinkFast)

![ 500](../_assets/Pasted%20image%2020260605103029.png)

* The RSTP **alternate** port role is a *discarding port* that receives a superior BPDU from another switch.
* This is the same as what you've learned about **blocking** ports in classic STP.
* An alternate port functions as a **backup** to the root port. If the root port fails, the switch can immediately move its best alternate port to forwarding.

![ 500](../_assets/Pasted%20image%2020260605103356.png)

* If the root port on SW3 fails, the alternate port can immediately transitions to forwarding port and become the root port **without entering transitional states**.
* This immediate move to forwarding state functions like a classic STP optional feature called **UplinkFast**. Because it is built into RSTP, you do not need to activate UplinkFast when using RSTP/Rapid PVST+.

---

### RSTP: BackboneFast Functionality

![ 500](../_assets/Pasted%20image%2020260605103852.png)

* One more STP optional feature that was built into RSTP is **BackboneFast**.

![ 500](../_assets/Pasted%20image%2020260605104009.png)

* Let's go back to classic STP. Let's say SW2-SW1 connection is cut off, so SW2 stopped receiving BPDUs from the root bridge.

![ 500](../_assets/Pasted%20image%2020260605104110.png)

* SW3 will then assume it is the root bridge, so it will start sending its own BPDUs to SW3.

![ 500](../_assets/Pasted%20image%2020260605104231.png)

* Since SW3 receives BPDUs from both SW1 and SW2, it will ignore SW2's BPDU because of its inferior bridge ID.
* SW3 will start advertising SW1's BPDU to SW2.
* SW2 will then accept SW1's BPDU and will start updating its STP topology and change the root port.

**What is BackboneFast?**

* With regular STP, SW3's **non-designated** interface would take at least 30 seconds before it transitions to forwarding state before it can forward SW1's BPDUs to SW2 to fix the STP topology.
* **BackboneFast** allows SW3 to expire the max age timers on its interface and rapidly forward the superior BPDUs to SW2.
* This functionality is built into RSTP, so it does not need to be configured.

---

### UplinkFast/BackboneFast Summary

* **UplinkFast** and **BackboneFast** are two optional features in classic STP. They must be configured to operate on the switch (not necessary to know for CCNA).
* Both features are built into RSTP, so you do not have to configure them. They operate by default.
* You do not need to have a detailed understanding of them for the CCNA. Know their names and their basic purpose (to help blocking/discarding ports rapidly move to forwarding).
* If you want to learn more, do a Google search for 'spanning tree uplinkfast' or 'spanning tree backbonefast'.

### RSTP Port Roles: Backup

![ 500](../_assets/Pasted%20image%2020260605110151.png)

* The RSTP **backup** port role is a discarding port that receives a superior BPDU from <ins>another interface on the same switch</ins>.
* This only happens when two interfaces are connected to **same collision domain** (which is only possible via a hub).

![ 500](../_assets/Pasted%20image%2020260605110600.png)

* When a BPDU is sent by SW2 through its designated port, the BPDU will then be flooded by a hub.

![ 500](../_assets/Pasted%20image%2020260605110706.png)

* As you can see above, SW2 will receive the same BPDU it sent in another interface because it is flooded by the hub.
* The interface receiving the same BPDU the another interface sent from same collision domain will become the **backup port** instead of **alternate port**.
* However, as you already know, ***Hubs are not used in modern networks***, so you will probably not encounter an RSTP backup port.
* Backup ports funcion as a backup for a designated port.
* As for who will be designated and backup, the interface with the **lowest port ID** will be selected as the designated port, and the other will be the backup port.
* **NOTE:** Hubs don't participate in STP, and only forward frames it receives, it does not add costs to BPDUs.

### RSTP Configuration/Verification in CLI

![ 500](../_assets/Pasted%20image%2020260605195225.png)

![ 500](../_assets/Pasted%20image%2020260605195238.png)

* To enter Rapid STP (Rapid PVST+ in Cisco switches) mode, use the command `spanning-tree mode rapid-pvst`
* Usually, you don't have to configure the STP mode manually as most Cisco switches nowadays uses Rapid PVST+ as a default.
* In the `show spanning-tree` output, you can see that the switch runs in Rapid STP mode (`Spanning tree enabled protocol rstp` output).
* SW3's `Gi0/1` interface has `Back` role, which means a **backup** port. The status says `BLK` (blocking port) but remember that in Rapid STP, this means **discarding** port.

![ 500](../_assets/Pasted%20image%2020260605195805.png)

* In SW4 spanning tree information, the `Gi0/0` interface `Role` is `Altn`. This means the interface is an **alternate port** which is in a discarding state.
* The `Sts` (Status) also says `BLK`. Remember that in Rapid STP `BLK` (blocking state) means **discarding** state.

#### Connecting Rapid STP-enabled switch to Classic STP-enabled switch

* Rapid STP **is** compatible with Classic STP.
* The interface(s) on the Rapid STP-enabled switch connected to the Classic STP-enabled switch will operate in Classic STP mode (timers, blocking -> listening -> learning -> forwarding process, etc).

![Pasted image 20260605200224.png](../_assets/Pasted%20image%2020260605200224.png)

* Let's say that **SW4** is in **Classic STP** mode. All the other switches have **Rapid STP** enabled.
* In this case, SW3's `G0/1` and `G0/0` interface, and SW2's `G0/1` interface which is connected to SW4 will operate in **Classic STP** mode.
* The remaining interfaces connected to **SW1** will operate in **Rapid STP** mode.

### RSTP BPDU (Wireshark packet capture)

![Pasted image 20260605200644.png](../_assets/Pasted%20image%2020260605200644.png)

* In `Protocol Version Identifier`, remember that for Classic STP, the value is **0** and for Rapid STP, the value is **2**.
* In `BPDU Type`, remember that the value for Classic STP is `0x00` and `0x02` for Rapid STP

![Pasted image 20260605200948.png](../_assets/Pasted%20image%2020260605200948.png)

* In Classic STP, it only uses the **first** bit and **last** bit (2 bits) of the 8 bits of the `BPDU flags`.
* Rapid STP uses **all** 8 bits. This flags are used for negotiation process that allow Rapid STP to converge much faster than Classic STP.

#### Another big difference between STP and Rapid STP (Wireshark)

* In classic STP, only the root bridge originated BPDUs, and other switches just forwarded the BPDUs they received.
* **In rapid STP, ALL switches originate and send their own BPDUs from their designated ports.**

---

### RSTP Hello timer, Aging

* All switches running Rapid STP send their own BPDUs every hello time (**2 seconds**).
* Switches 'age' the BPDU information much more quickly. In classic STP, a switch waits 10 hello intervals (20 seconds). In rapid STP, a switch considers a neighbor lost if it misses 3 BPDUs (6 seconds). It will then 'flush' all MAC addresses learned on that interface.

For PC1 to learn PC2's MAC address again, it will go through the process of normal broadcasting and flooding ARPs. The the frames sent by PC1 will go through the learned paths to reach PC2.

### RSTP Link Types

RSTP distinguishes between three different 'link types':

* **Edge:** A port that is connected to an end host. Moves directly to forwarding, without negotiation.
  * Remember **PortFast Edge**? Well, the mechanism is exactly the same! Now in RSTP, we have **PortFast**, **UplinkFast**, and **BackboneFast** built in!
* **Point-to-point**: a direct connection between two switches (full-duplex connection).
* **Shared:** a connection to a hub. Must operate in half-duplex mode.

---

### RSTP Link Types: Edge

![ 500](../_assets/Pasted%20image%2020260605204426.png)

* Edge ports are connected to end hosts.
* Because there is no risk of creating a loop, they can move straight to the forwarding state without the negotiation process.
* They function like a classic STP port with PortFast enabled.
* The same command is also used to enable them: `spanning-tree portfast`

---

### RSTP Link Types: Point-to-Point

![ 500](../_assets/Pasted%20image%2020260605204712.png)

* Point-to-point ports connect directly to another switch.
* They function in **full-duplex**
* You don't need to configure the interface as point-to-point (it should be connected).
* To manually enable it however, use this command: `spanning-tree link-type point-to-point`

---

### RSTP Link Types: Shared

![ 500](../_assets/Pasted%20image%2020260605205001.png)

* Shared ports connect to another switch (or switches) via a hub.
* They function in **half-duplex**.
* You don't need to configure the interface as shared (it should be detected).
* To manually enable it, use this command: `spanning-tree link-type shared`
