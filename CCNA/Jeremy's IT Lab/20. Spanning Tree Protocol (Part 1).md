### Network Redundancy

* Redundancy is an essential part of network design.
* Modern networks are expected to run 24/7/365. Even a short downtime can be disastrous for a business.
* If one network component fails, you must ensure that other components will take over with little or no downtime.
* As much as possible, you must implement redundancy at every possible point in the network.

### Layer 2 Loops (Broadcast Storms)

![ 500](../_assets/Pasted%20image%2020260527083935.png)

**Spanning Tree Protocol** (STP) works at Layer 2 and is used for allowing network redundancy. It handles issues such as **broadcast storms** that occur in Layer 2. Without STP, the topology above will **have some issues**.

**What is broadcast storm?**

![ 500](../_assets/Pasted%20image%2020260527084203.png)

Here is the example we will use to demonstrate what are broadcast storms. Let's say PC1 wants to communicate with PC2. PC1 will have to know PC2's MAC address first, so it will send and ARP request.

![ 500](../_assets/Pasted%20image%2020260527084420.png)

As we learned before, when switches receives a broadcast frame, it will flood it to all interfaces except where it received the broadcast frame. Eventually, PC2 will receive the broadcast and will send a known unicast frame (ARP reply). However, there will be a problem even after PC1 receives the ARP reply!

![ 500](../_assets/Pasted%20image%2020260527084706.png)

Look at the **boxed broadcast frames**. The problem is the broadcasting will still persist *indefinitely* even after PC2 sent an ARP reply due to the nature of how broadcast frames are handled by switches!

In Layer 3 packets, there is a concept called **Time To Leave (TTL)** to prevent such loops where it will drop the packet if TTL reaches 0. **In Layer 2, however, there is no concept of TTL**. So the broadcast frames will flood the interface infinitely and will eventually clog the traffic! This is called a **broadcast storm**.

Let's demonstrate how will the highlighted broadcast frames will start an infinite loop and cause a **broadcast storm**:

![ 500](../_assets/Pasted%20image%2020260527085148.png)
![ 500](../_assets/Pasted%20image%2020260527085222.png)
![ 500](../_assets/Pasted%20image%2020260527085245.png)

Fast forward, the network will look like this *indefinitely*:

![ 500](../_assets/Pasted%20image%2020260527085435.png)

Broadcast storm isn't the only problem. There will be **MAC address flapping** because source MAC address repeatedly arrive on different interfaces and switches have to continuously update its MAC address table!

**MAC Address Flapping:**

* Network congestion isn't the only problem. Each time a frame arrives on a switchport, the switch uses the source MAC address field to 'learn' the MAC address and update its MAC address table. When frames with the same source MAC address repeatedly arrive on different interfaces, the switch is continuously updating the interface in its MAC address table. This is known as **MAC Address Flapping**.

### Spanning Tree Protocol (STP) Intro

* 'Classic Spanning Tree Protocol' is IEEE 802.1D
* Switches from ALL vendors run STP by default.
* STP prevents Layer 2 loops by placing redundant ports in a blocking state, essentially disabling the interface.
* These interfaces act as backups that can enter a forwarding state if an active (=currently forwarding) interface fails.
* Interfaces in a forwarding state behave normally. They send and receive all normal traffic.
* Interfaces in a blocking state only send or receive STP messages (called BPDUs = Bridge Protocol Data Units)

Let's talk about 'Bridge' in BPDUs first.

### Hubs, bridges, and switches

![ 500](../_assets/Pasted%20image%2020260527090734.png)

Before switches replaced hubs (which doesn't know what MAC address are and only forwards any received frames), there was a networking device that was introduced before switches, called **bridge**.

Spanning Tree Protocol still uses the term 'bridge'. **However, when we use the term 'bridge', we really mean 'switch'**. Bridges are not used in modern networks.

**Spanning Tree Protocol deep dive:**

* By selecting which ports are **forwarding** and which ports are **blocking**, STP creates a single path to/from each point in the network. This prevents Layer 2 loops.
* There is a set process that STP uses to determine which ports should be forwarding and which should be blocking.
* STP-enabled switches send/receive Hllo BPDUs out of all interfaces, the default timer is **2 seconds** (the switch will send a Hello BPDU out of every interface, once every 2 seconds).
* If a switch receives a Hello BPDU on an interface, it knows that interface is connected to another switch (routers, PCs, etc. o not use STP, so they do not send Hello BPDUs).

![ 500](../_assets/Pasted%20image%2020260527092531.png)

Hello BPDU is used by switches to advertise themselves to other switches and also to learn other switches.

### STP BPDUs, Root Bridge election

* Switches use one field in the STP BPDU, the **Bridge ID** field, to elect a **root bridge** for the network.
* The switch with the lowest **Bridge ID** becomes the **root bridge**.
* ALL ports on the **root bridge** are put in a forwarding state, and other switches in the topology must have a path to reach the root bridge.

Traditionally, the Bridge ID of a STP BPDU look like this:

![ 500](../_assets/Pasted%20image%2020260527093126.png)

* The default bridge priority is **32768** on all switches, so by default the MAC address is used as the tie-breaker (lowest MAC address becomes the root bridge).
* ***The Bridge Priority is compared first. If they tie, the MAC address is then compared.***

All of the bridge ID are set to default priority (32768). So we have to compare MAC addresses and pick the lowest MAC address (based on hex value) to elect the forwarding state.

### PVST, Extended System ID

![ 500](../_assets/Pasted%20image%2020260527094936.png)

The bridge priority field is updated into two parts: **Bridge Priority** and **Extended System ID (which is the VLAN ID)**.

Cisco switches use a version of STP called **PVST** (Per-VLAN Spanning Tree). PVST runs a separate STP 'instance' in each VLAN, so in each VLAN different interfaces can be forwarding/blocking.

![ 500](../_assets/Pasted%20image%2020260527095446.png)

You may have wondered why the default priority is set to 32768. Well, it is because the Bridge Priority field is 16 bits in length, and the most significant bit is set to 1 by default. So the Bridge Priority WAS 32768.

However with addition of Extended System ID, where we add the VLAN ID, it changed. The default value of VLAN ID is 1. Therefore, in extended version of bridge priority, the default value is set to **32769** (32768 + 1).

If you want to change the switch's bridge priority (without changing the VLAN numbers), what is the minimum unit of increase/decrease?

![ 500](../_assets/Pasted%20image%2020260527095830.png)

As you can see, we can only change the bridge piority in units of 4096. What if we reduce it's value?

### Root Bridge (continuation)

![ 500](../_assets/Pasted%20image%2020260527100151.png)

By using the Extended System ID, the default Bridge Priority is set to 32769. Note that the topology above is in a single VLAN. If you have multiple VLANs, the default Bridge Priority in VLAN2 will be 32770, and the default Bridge Priority in VLAN3, will be 32771, and so on...

Each VLAN, the root bridge will be different, and the designated ports will also be different. Since SW1 have the lowest MAC address in this VLAN, it will serve as the root bridge.

***Remember, all interfaces on the root bridge are designated ports. Designated ports are in a forwarding state.***

**Few notes about the root bridge:**

* When a switch is powered on, it assumes it is the root bridge.
* It will only give up its position if it receives a *superior* BPDU (lower bridge ID).
* Once the topology has converged and all switches agree on the root bridge only the root bridge sends BPDUs.
* Other switches in the network will forward these BPDUs, but will not generate their own original BPDUs.

### STP Port Role Selection (Root Port Selection via Root Cost)

1. The switch with the lowest bridge ID is elected as the root bridge. All ports on the root bridge are **designated ports** (forwarding state).
1. Each remaining switch will select ONE of its interfaces to be its **root port**. The interface with the lowest *root cost* will be the root port. Root ports are also in a forwarding state.

### Root Cost

![ 500](../_assets/Pasted%20image%2020260527151213.png)

Each interface have an associated spanning tree cost. The root bridge is a special one such that each of its interface will have a root cost **0**.

![ 500](../_assets/Pasted%20image%2020260527151744.png)

All of the interfaces are Gigabit Ethernet ports, so each interface will have a root cost of 4. The **root cost** is the total cost of the outgoing interfaces along the path to the root bridge. SW1 is the root bridge, so it has a cost of 0 on all of its interfaces.

They are Gigabit Ethernet interfaces, but you don't count the cost of the receiving interface, just the sending/outgoing interface.

![ 500](../_assets/Pasted%20image%2020260527152626.png)

SW1 advertises its root cost of 0 into its BPDUs.

![ 500](../_assets/Pasted%20image%2020260527152732.png)

SW2 will receive the BPDU, and add the cost of its outgoing interface G0/1, which is 4 when it it floods the BPDUs out of its interfaces.

![ 500](../_assets/Pasted%20image%2020260527153110.png)

SW3 will do the same.

So which interfaces will switches choose as its **root port**?

![ 500](../_assets/Pasted%20image%2020260527153655.png)

SW2 will choose `G0/1` as its root port

![ 500](../_assets/Pasted%20image%2020260527153614.png)

SW3 will choose `G0/0` as its root port.

**NOTE:** The ports connected to another switch's root port MUST be **designated**. Because the root port is the switch's path to the root bridge, another switch must not block it. In the topology above the connected ports to root ports are already designated.

### STP Port Role Selection (Root Port Selection via Neighbor Bridge ID)

1. One switch is elected as the root bridge. All ports on the root bridge are **designated ports** (forwarding state). Root bridge selection:
   1. **Lowest bridge ID**
1. Each remaining switch will select ONE of its interface to be its **root port** (forwarding state). Ports across from the root port are always **designated** ports. Root port selection:
   1. **Lowest root cost**
   2. **Lowest neighbor bridge ID**
   ![ 500](../_assets/Pasted%20image%2020260527155017.png)
   SW1 will use it's `G0/0` interface as its root port and SW4 will use it's `G0/1` interface as its root port. In SW3, both the total cost of the interface are tiebreaker (Cost = 8), so SW3 will pick the neighbor with the lowest MAC address, which is SW1. SW3's `G0/0` interface is connected to SW1 so it will be the root port, and SW1's `G0/1` interface will be a designated port because it is connected to SW3's root port.
   3. **Lowest neighbor port ID**
   ![ 500](../_assets/Pasted%20image%2020260527161238.png)
   STP Port ID = port priority (default 128) + port number
   So if both the lowest root cost and lowest neighbor bridge ID are tie breaker, the switch will pick the port with the lowest STP port ID.
   ![ 500](../_assets/Pasted%20image%2020260527163156.png)
   On SW3, the `G0/2` interface is selected as the root port. Why not `G0/0`? **The NEIGHBOR switch's port ID is used to break the tie, not the local switch's port ID.**

### Blocking Ports to prevent loops

![ 500](../_assets/Pasted%20image%2020260527163321.png)

All that's left is the connection between SW2 and SW3, and the rest of the active ports are in forwarding state. So, do we need to block both the `G0/0` interface of SW2 and `G0/1` interface of SW3? The answer is ***no***.

Remember, ***Every collision domain has a single STP designated port***. Unlike hubs, every connection between switches and other devices is a single collision domain.

![ 500](../_assets/Pasted%20image%2020260527163756.png)

If you ignore the red-boxed collision domain, have you noticed that all of the other collision domains have at least *one designated port*? So in connection between SW2 and SW3, how do we determine which port to use as a forwarding state?

To pick which port to determine as a forwarding state:

1. The switch with the lowest root cost will make its port designated.
1. If the root cost is the same, the switch with the lowest bridge ID will make its port designated.
1. The other switch will make its port non-designated (blocking)

![ 500](../_assets/Pasted%20image%2020260527164504.png)

Since both SW1 and SW2 have the same root cost, we will have to pick which has the lower bridge ID. SW2 have the lower bridge ID, so we will pick the port on SW2. SW3's `G0/1` interface will be **non-designated** which means it is in **blocking state**. It will block the port to prevent Layer 2 loops.

### STP Process Summary

1. One switch is elected as the root bridge. All ports on the root bridge are **designated ports** (forwarding state). Root bridge selection:
   1. Lowest bridge ID
1. Each remaining switch will select ONE of its interfaces to be its **root port** (forwarding state). Ports across from the root port are always **designated** ports.
   Root port selection:
   1. Lowest root cost
   2. Lowest neighbor bridge ID
   3. Lowest neighbor port ID
1. Each remaining collision domain will select ONE interface to be a **designated port** (forwarding state). The other port in the collision domain will be **non-designated** (blocking)
   Designated port selection:
   1. Interface on switch with lowest root cost
   2. Interface on switch with lowest bridge ID
