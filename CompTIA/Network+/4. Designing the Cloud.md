Cloud changed the landscape of technology. The way we do things change drastically with introduction of cloud:

* With a click of a button, we can deploy an application and multiple services all seemingly with unlimited resources.
* We can scale up or down these services (elasticity), conforming to demands
* Applications also scale, and we are able to access them anywhere in the world
* For efficiency and cost management, many different clients can use the same cloud infrastructure. This is also called as \*\*multitenancy

### Virtual networks

* We might have 100 individual servers that are running in a data center (designed with network redundancy, multiple load balancers, firewalls, etc.)
* We can move this infrastructure to the cloud by migrating 100 physical servers to one physical server! That physical server will have 100 *virtual servers* inside it.
* Now that we have one big physical server with 100 virtual servers, what will we do with the network infrastructure? We will virtualize the network infrastructure as well! This is called **Network function virtualization (NFV)**

### Network function virtualization (NFV)

![Pasted image 20260423212611.png](../../_assets/Pasted%20image%2020260423212611.png)

* Replace physical networking devices (routers, switches, firewalls, etc.) with virtual versions
* They provide the same functionality as a physical device (routing, switching, load balancing, firewalls, etc.). We can see and manage these functionalities from a **hypervisor**
* We can easily deploy network functions (create new firewalls, switch, router configs, etc.) by a single click
* This makes NFV be flexible for many deployment options (virtual machine, container, fault tolerance, etc.)

### Connecting to the cloud (VPCs)

![Pasted image 20260423215748.png](../../_assets/Pasted%20image%2020260423215748.png)

* A common application instance inside cloud infrastructure might include web server, db server, load balancers, virtual switches, virtual routers, virtual firewalls, etc. All of this will run inside a \*\*Virtual Private Cloud (VPC)
* In larger environments, it is very common to have separate VPCs for different application instances or different parts of company.
* A **transit gateway** is used to connect all of VPCs together, effectively creating a "cloud router"
* To connect to VPCs from the outside, you may need to add additional security in the form of **VPN connection**. The VPN will connect to transit gateway allowing you to communicate with different VPCs.
